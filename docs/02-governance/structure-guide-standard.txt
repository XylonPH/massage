Title: Massage Nexus PHP Structure Guide Standard
Document Version: 1.40
Date Created: 2026-07-21
Revision Date: 2026-07-21

Document Description: Defines the required purpose, metadata, timestamps, field layers, ordering, validation, and maintenance workflow for PHP-readable database structure guides under data/structure_guide/.

================================================================================
1. PURPOSE AND AUTHORITY
================================================================================

PHP structure guides are reviewable field-level descriptions of proposed or working database
records. They are documentation expressed as PHP arrays. They are not migrations, seeds,
runtime configuration, or proof that a proposed structure has been accepted or deployed.

They live under data/structure_guide/ because they are machine-readable repository source
artifacts consumed by validation and field-index tooling. Their location does not make them
runtime data, import datasets, finalized contracts, or production schemas.

docs/04-architecture/database-structure.txt remains authoritative for database and collection
boundaries. Current feature specifications govern feature meaning. Structure guides preserve
the matching record shape and field semantics.

================================================================================
2. REQUIRED GUIDE METADATA
================================================================================

Each detailed PHP structure guide begins with one comment block that identifies:
- Title: a readable title that is not merely the raw filename;
- Version: the version of that individual structure-guide file;
- Collection: the exact collection or table name, matching the snake_case filename;
- Description: a short explanation of what one record in the collection represents;
- Purpose: why the PHP-readable file exists and how it should be used.

Write this metadata as a single leading PHP doc comment (`/** ... */`) with each metadata line
starting with `*`, immediately after the opening `<?php` tag. Validation tooling matches this exact
comment style; metadata written as `//` line comments or placed elsewhere in the file will fail
validation even if the same information is present.

Do not include an Author field. Git records authorship, and repeating the same author in every
file adds no structural meaning.

Every accepted saved revision updates the file version. The Version value must strictly follow the
major.two-digit-minor format (e.g., 1.00 or 1.10). Use the size of the change to select the
increment:
- +0.01 for a non-semantic correction to wording, comments, formatting, or examples;
- +0.10 for accepted field, default, order, embedded shape, index, boundary, relationship,
  validation, or other structural guidance;
- +1.00 for a fundamental redesign that replaces the governing record model.

An undiscussed or exploratory collection may begin below 1.00. A structure discussed and accepted
as the governing direction may begin at or be promoted to 1.00. Do not lower an existing version.

================================================================================
3. GUIDE TIMESTAMPS
================================================================================

$created_at records the exact UTC date and time when the guide file was first created and must not
change during later edits. A new guide sets $updated_at to the same value as $created_at.

$updated_at records the exact UTC time of the latest accepted guide revision. Every saved change
to the file updates this value to the actual current UTC time.

Use ISO 8601 UTC format with seconds:

YYYY-MM-DDTHH:MM:SSZ

Do not use midnight, a date-only approximation, local time with a Z suffix, or the time of a
previous revision unless that is the exact UTC edit time. Obtain the current UTC value from the
system clock. The timestamp variables describe the guide revision; similarly named fields in a
sample record remain illustrative record data.

================================================================================
4. REQUIRED VARIABLES AND ORDER
================================================================================

For a collection named article_main, use this order and naming pattern:
1. $created_at and $updated_at;
2. $article_main_default and $multilingual_text_sample, in either order, with the multilingual
   sample present only when at least one field is multilingual;
3. $article_main;
4. $article_main_field_order;
5. $article_main_embedded_structure;
6. $article_main_field_property;
7. $article_main_subfield_property;
8. $article_main_legacy_name_map, only when an actual legacy structure requires mapping;
9. $article_main_index_list;
10. $article_main_boundary;
11. one final return statement.

Replace article_main with the exact collection name in every collection-specific variable.
Required arrays remain present even when they are currently empty, except for the explicitly
conditional multilingual sample and legacy map.

Validation tooling mechanically checks declaration order only from the sample record onward: the
sample, field order, embedded structure, field-property map, subfield-property map, index list, and
boundary must appear in that relative sequence. The timestamps, *_default, and
$multilingual_text_sample are expected before the sample record by convention but are not currently
order-checked by tooling; place them first regardless, since a future validator revision may enforce
the full sequence.

The collection variable, such as $article_main, is one fully populated sample record. It lists
every possible top-level stored property using a representative sample value. Each property has a
short inline comment explaining its meaning. A document-store record may omit sparse or
non-applicable properties, but the sample variable must not omit a possible property.

The *_default array contains the authoritative values implied when the corresponding stored
property is absent. It contains only fields with real omission defaults; it is not a complete
sample record. Application queries, filtering, conditional formatting, validation, and display
logic must treat an omitted property as its documented default. For example, when a field defaults
to "test", a filter for that field equal to "test" must include records where the field is omitted.

The *_field_order array lists every top-level stored property exactly once in intended logical
order. Validation tooling enforces a strict key-equality check: the keys in the sample record,
*_field_order, and *_field_property must match exactly. Furthermore, every key inside *_default
must exist in the sample record.

The *_embedded_structure array shows the complete shape of each embedded object or list item. If
populated, validation tooling strictly requires *_subfield_property to be populated as well.
The *_field_property array describes every top-level property's data contract. The
*_subfield_property array describes every nested property using an unambiguous dotted path or
another clearly documented path notation. Keep stored defaults separate from field-definition
metadata.

The sample, field order, field-property map, embedded structure, subfield-property map, index
list, boundary, and final return map must be written explicitly. Do not derive these structures
with array_keys(), foreach, compact(), inferred primitive types, generated labels, or generated
descriptions. Explicit declarations are intentional review surfaces: a reviewer must be able to
inspect each field's meaning and contract without executing the guide or reverse-engineering a
template.

Every top-level and nested field-property entry must explicitly provide:
- field_label;
- field_description, describing the field's actual meaning rather than repeating its name;
- type_data;
- type_field.

Add the applicable contract attributes when they carry meaning, including type_sql,
is_mandatory, is_relational, is_indexed, is_unique, is_translatable, visibility_scope,
default_value, length or numeric limits, accepted formats, and validation constraints. Length and
numeric limits use these canonical keys: min_character and max_character for text length,
min_number and max_number for numeric range. Accepted-format constraints use
constraint_text_input (e.g., ['URL'], ['EMAIL']). Do not invent alternate key names such as
character_limit or max_length for these same concepts. Do not infer a field's type from its sample
value: null is often only an illustrative absence and does not identify the stored type.

Field properties use standardized abbreviation codes. type_data denotes the primitive type (e.g.,
S for String, I for Integer, B for Boolean, A for Array, O for Object). type_field denotes the
semantic interface format (e.g., TXT for Text, JSE for JSON/Embedded, DDL for Dropdown/Enum,
TGL for Tag List, CBL for Checkbox List, CHK for Checkbox, NMB for Number, DTS for DateTime
String, DTP for Date Picker). This list is illustrative, not exhaustive: before introducing a new
type_data or type_field code, search existing structure guides for a code that already covers the
same concept and reuse it, the same way field names are checked against data/field_index.txt
before a new one is added.

The $multilingual_text_sample uses eng as its example language. It is strictly required by
validation tooling if any field defines 'is_translatable' => true; otherwise, it is optional.
Do not add another language merely to make the generic sample appear multilingual. Actual
collection sample records may demonstrate additional languages when useful.

The optional *_legacy_name_map exists only for a collection derived from a legacy database or
schema. New Massage Nexus collections do not include an empty or speculative legacy map. When
present, it is a flat map from each old field name to its current replacement name, or to a short
note when the replacement is conditional or split across more than one current field:

$field_main_legacy_name_map = [
    'field_title' => 'field_label',
    'type_validation' => 'validation_rule_list or constraint_text_input, depending on meaning',
];

Every *_index_list entry uses this shape:

[
    'index_key' => 'article_slug_unique',
    'index_name' => 'uq_article_main_article_slug',
    'type_index' => 'STD',
    'is_unique' => true,
    'is_sparse' => false,
    'index_field_list' => [
        [
            'field_name' => 'article_slug',
            'type_index_mode' => 'ASC',
            'sort_order' => 10,
        ],
    ],
    'sort_order' => 10,
];

index_key is the stable guide key. index_name is the intended physical database index name.
type_index describes the overall index kind. Common values include STD (Standard), 2DSPHERE or 2D
(Geospatial), TTL (Expiring), TEXT (Search), or MULTIKEY. type_index_mode describes how one field
participates in the index. Collection-level index definitions belong here rather than only in
relationship metadata.

The index entry carries two distinct sort_order values. The entry-level sort_order (the one
alongside index_key, index_name, and type_index) orders this index entry relative to the other
entries in *_index_list. The sort_order nested inside each index_field_list item orders that
field's position within its own compound index; compound fields use increments of 10 for this
nested value.

The *_boundary array identifies what the collection owns, which other collections or datasets it
references, and what responsibilities remain outside it. Boundary entries must distinguish a
stored reference from embedded ownership and must not invent unresolved relationships. Use exactly
these three keys:

$article_main_boundary = [
    'owns' => ['the article_main record fields and embedded structures documented in this file'],
    'reference_field_list' => ['language_original_id', 'author_user_id_list'],
    'does_not_own' => ['records stored in referenced collections', 'runtime authorization, migration, seeding, or deployment behavior'],
];

reference_field_list names the actual stored fields that hold an identifier into another
collection (a foreign-key-style reference). It does not include taxonomy-coded fields such as a
type_* or status_* select: a taxonomy option code is owned classification data validated against a
taxonomy JSON file, not a reference to another collection's record. When a collection owns no
cross-collection references, use an empty array rather than omitting the key.

The file ends with one return statement containing every declared structural array under a key
matching its variable name. Include $multilingual_text_sample and *_legacy_name_map only when they
exist. Do not return a different anonymous schema in place of the named variables.

Do not introduce a sample-only field that is absent from field order and field properties.
Embedded records must have a stable documented shape when their members affect application
behavior.

Preserve established field names and semantics. Before introducing any new field name,
including a nested field, check data/field_index.txt and follow AGENTS.md taxonomy and field
governance. The indexing tool flattens all nested paths into bare field names. This guarantees
that a field name carries the exact same semantic contract across the entire database, whether at
the root level or nested deeply. Reuse a field only when its existing meaning matches.

================================================================================
5. CHANGE WORKFLOW
================================================================================

For a structure change:
1. read the affected feature specification and database boundary documentation;
2. inspect data/field_index.txt before choosing a field name;
3. update the relevant structure guide before implementing dependent application code;
4. update the guide version and exact UTC $updated_at value;
5. update defaults, sample, field order, embedded structure, field properties, subfield
   properties, index list, boundary, return statement, and comments consistently;
6. run php tools/script/build_field_index.php when a guide adds, renames, or removes a field;
7. run PHP syntax validation for every modified guide;
8. run php tools/script/validate_structure_guides.php;
9. inspect the generated index diff and the final guide diff.

Add or update taxonomy JSON in the same change when a new field has controlled or enumerated
options. Do not create taxonomy entries for free text, identifiers, timestamps, or relationships
that have no controlled option set.

================================================================================
6. COMPATIBILITY AND IMPLEMENTATION
================================================================================

Structure revisions should be additive and recoverable when existing data may already use an
older shape. Document compatibility meaning where legacy fields remain temporarily supported.

Application code must continue to validate input, authorize mutations, normalize embedded
records, and constrain relationship identifiers. In Laravel implementations, Eloquent Models
corresponding to a structure guide should define sparse omission defaults inside their `$attributes`
model array or accessor methods to ensure runtime queries match the guide's `*_default`
specifications. A PHP guide does not replace runtime security, validation, migrations, data
backfills, tests, or rollout planning.
