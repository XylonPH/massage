Title: Massage Nexus AI-Assisted Development Workflow
Document Version: 1.50
Date Created: 2026-07-16
Revision Date: 2026-07-19
Document Description: Defines how repository coding agents and human-directed AI assist Massage Nexus planning, documentation, implementation, testing, review, debugging, and maintenance while following current project authority, preserving user work, limiting context, implementing authorized changes directly, and validating results in proportion to risk. Runtime AI features inside Massage Nexus are governed separately by the Runtime AI System and the applicable capability specifications.

Document Relationships:
AGENTS.md is the active operational instruction file for repository agents. Shared Project
Standards defines reusable project rules. Documentation Writing and Management Standard owns
documentation structure and versioning. Testing and Quality Assurance Strategy owns the
cross-project testing model. GitHub Development and Repository Operations owns repository
hosting and collaboration rules. Runtime AI System governs AI features executed by the
Massage Nexus application. This list is intentionally limited to direct governing
relationships rather than attempting to inventory all documents in docs.

================================================================================
1. PURPOSE AND BOUNDARY
================================================================================

This document governs AI-assisted work on the Massage Nexus repository.

It applies to:
- planning a focused repository change;
- inspecting project documentation, code, tests, and data definitions;
- implementing authorized changes;
- reviewing, debugging, refactoring, and testing;
- updating authoritative project documentation when accepted behavior changes;
- explaining technical decisions at a level the user can evaluate.

It does not define the product behavior of runtime AI features used by visitors, users,
providers, contributors, moderators, support staff, or administrators. Those features follow
Massage Nexus Runtime AI System and the specification that owns the affected capability.

AI should reduce the user's work rather than create repeated review cycles caused by generic,
incomplete, overly long, or poorly organized output.

The preferred result is:
- specific to Massage Nexus;
- complete for the requested scope;
- compact where explanation is unnecessary;
- explicit about relevant constraints and edge cases;
- tested and validated in proportion to risk;
- honest about unresolved decisions or limitations;
- free from unsolicited plans, handoffs, implementation-note files, and generic best-practice
  material.

================================================================================
2. AUTHORITY AND CONFLICT HANDLING
================================================================================

Use this authority order:

1. current explicit user decision;
2. current project-specific specification;
3. current Shared Project Standards;
4. current code and tests, where they describe implemented behavior;
5. older project documents;
6. historical or archived material;
7. general conventions, assumptions, or generic AI knowledge.

AGENTS.md supplies the repository's active working instructions and documentation routing. A
closer scoped AGENTS.md applies where one exists.

Project-specific specifications may override Shared Project Standards only through an
explicitly recorded project exception. An inherited rule that merely became outdated must be
synchronized rather than treated as an intentional exception.

When implementation and the governing specification disagree:
- do not silently assume that either one is correct;
- determine whether the implementation is wrong, the documentation is stale, or a decision is
  unresolved;
- fix both when the requested and accepted result is clear;
- request direction when choosing one would create a new product decision or destructive
  consequence.

Archived documents and Working Notes are not current implementation authority.

================================================================================
3. TASK ENTRY AND CONTEXT SELECTION
================================================================================

Before changing files:
1. inspect Git status and preserve unrelated user changes;
2. read AGENTS.md and any applicable scoped instructions;
3. locate the smallest sufficient set of current specifications;
4. inspect the affected code, tests, configuration, and structured data;
5. confirm that the requested behavior is not already implemented;
6. identify relevant data, security, privacy, localization, accessibility, route, and
   compatibility effects;
7. identify any unresolved decision that would materially change the result.

Do not load the entire documentation tree by default.

Prefer:
- exact filename and symbol searches;
- the documentation routing in AGENTS.md;
- the governing capability specification;
- relevant ranges of large files;
- affected code and tests;
- data/field_index.txt before proposing a new field;
- matching taxonomy and structure-guide sources when structured data is affected.

Expand context only when a dependency requires it, a conflict appears, the task crosses
several capabilities, or the user requests a broader review.

Do not recursively read archives, Working Notes, large idea catalogues, or unrelated feature
folders.

No separate documentation manifest is currently required. AGENTS.md, semantic docs folders,
repository search, and data/field_index.txt provide the active routing mechanisms. A manifest
should be introduced only when document discovery becomes unreliable, and it should be
generated or easily maintained rather than becoming another manual authority.

================================================================================
4. IMPLEMENTATION AUTONOMY AND APPROVAL BOUNDARIES
================================================================================

An explicit request to build, modify, fix, or update authorizes ordinary reversible repository
work within that scope. The agent should implement and test the requested result directly
without asking for a second confirmation merely because AI produced the change.

Recommendations, exploratory architecture, unaccepted product choices, and runtime-generated
content remain proposals until accepted through the relevant workflow.

Stop and request direction when work requires:
- a missing product decision that would materially change behavior;
- destructive, irreversible, or production data action not already explicitly authorized;
- deletion or incompatible migration with unknown downstream effects;
- publication, deployment, release, commit, push, or external communication not requested by
  the user;
- purchasing, paid-service activation, or use of production credits outside an accepted
  workflow;
- access to unavailable secrets, accounts, evidence, or private data;
- a meaningful expansion beyond the requested systems or people.

Normal additive schema or code changes that are necessary for an explicitly requested feature
do not require a ceremonial approval step when the governing design is clear and the change is
recoverable. Destructive database, migration, or live-environment actions require exact target
validation and explicit authority.

Do not create formal task packets or implementation plans for simple, isolated work. A concise
in-conversation scope statement is sufficient. A durable task packet is appropriate only when
the user requests it or broad, delegated, high-risk, destructive, or multi-stage work genuinely
needs explicit boundaries and acceptance gates.

================================================================================
5. WORKING NOTES, HANDOFFS, AND IMPLEMENTATION RECORDS
================================================================================

Working Notes:
- belong to the user;
- may be incomplete, repetitive, inconsistent, or temporary;
- are not read, reorganized, edited, or treated as authority unless the user explicitly asks;
- remain outside ordinary repository implementation work.

AI Chat Handoffs:
- preserve continuity between tasks or AI systems when requested;
- do not replace current subject-specific specifications;
- are not created or updated automatically during ordinary repository work;
- normally remain outside the source repository.

Implementation records:
- are not generated merely to restate a completed change;
- are created only when the user requests one, a governing specification requires one, or the
  change genuinely needs a durable operational record beyond Git history and the final report;
- preserve final behavior, tests, migrations, risks, and limitations rather than a debugging
  transcript.

================================================================================
6. DOCUMENTATION BEHAVIOR
================================================================================

Before creating, moving, renaming, splitting, merging, or substantially editing documentation,
follow Massage Nexus Documentation Writing and Management Standard.

AI must:
- update the authoritative file rather than create a competing copy;
- preserve accepted requirements, constraints, examples, metadata, and edge cases;
- remove only obsolete, rejected, superseded, or truly duplicated material;
- identify unresolved matters instead of inventing acceptance;
- keep repository filenames lowercase, dash-separated, unnumbered, unversioned, and .txt
  unless an approved exception applies;
- update Document Version and Revision Date when required;
- preserve Date Created;
- update another document only when its meaning actually changed;
- avoid duplicating a rule owned by another specification;
- ensure the result can be understood without reconstructing the chat.

Documentation maturity, version numbering, metadata, file size, splitting, and public-document
rules are owned by the Documentation Writing and Management Standard and are not duplicated
here.

The roadmap records dependency and milestone order. The Simple Checklist records completion.
Update the roadmap only when an accepted milestone or dependency changes. Check off a feature
only when its accepted scope is implemented and the relevant validation passes; a document,
draft, or newly added test alone does not prove completion.

================================================================================
7. CODE, DATA, AND FILE RESPONSIBILITY
================================================================================

During implementation:
- remain within the requested scope;
- preserve unrelated user changes in a dirty worktree;
- follow the current framework and repository conventions;
- keep one coherent responsibility per file without fragmenting simple behavior;
- reuse established Models, Actions, Services, Policies, Requests, Resources, components,
  translation keys, and tests where appropriate;
- keep business logic out of views;
- enforce authorization and validation server-side;
- use translation keys for user-facing interface text;
- keep secrets and environment-specific values out of source, logs, prompts, tests, and
  documentation;
- use deterministic synthetic test data;
- avoid unrelated refactoring;
- avoid installing or upgrading dependencies unless required by the task;
- do not leave generic TODO comments instead of implementing requested logic unless the task is
  explicitly exploratory;
- keep changes narrow and recoverable.

File length is a review signal, not an automatic line-count rule. Refactor when a file mixes
responsibilities, forces unrelated context loading, or makes focused testing difficult. Do not
split cohesive logic merely to satisfy an arbitrary number of lines.

Before adding, renaming, or removing a field:
- check data/field_index.txt;
- inspect the matching taxonomy or structure-guide source;
- reuse an existing semantic field when it answers the same question;
- preserve stable identifiers and option meanings;
- regenerate the field index in the same change when required by AGENTS.md;
- assess compatibility, migration, defaults, indexes, and existing data.

Treat taxonomy work as structured design. Check code, label, description, inclusion, exclusion,
relationships, examples, hierarchy, and duplication. Do not add Other by default or combine
separate concepts merely to shorten a list.

================================================================================
8. TESTING AND VALIDATION
================================================================================

Testing follows AGENTS.md, Shared Project Standards, Testing and Quality Assurance Strategy,
and the affected capability specification.

In proportion to risk:
1. run the smallest relevant test or check while developing;
2. run the affected capability suite;
3. run relevant formatting, linting, static analysis, type, build, security, accessibility, or
   contract checks;
4. run the broader regression suite at an appropriate checkpoint;
5. manually verify important visual, interactive, assistive-technology, or operational behavior
   when automation is insufficient;
6. inspect the final diff and remove temporary debugging.

AI-generated tests are not the only evidence of correctness. Independently inspect edge cases,
failure behavior, authorization, validation, privacy, compatibility, and regressions. Do not
weaken existing tests merely to make a change pass.

Runtime AI tests use fake providers by default and must not spend production credits. Explicit
provider integration tests use approved non-production credentials and data, run separately,
and never expose secrets in output.

Summarize successful checks. Preserve detailed output only when a failure needs diagnosis.
Never claim a test passed unless it was actually run successfully.

================================================================================
9. SECURITY, PRIVACY, AND EXTERNAL SYSTEMS
================================================================================

AI-assisted development must not:
- expose passwords, hashes, keys, tokens, private evidence, personal data, or environment
  secrets;
- copy production data into prompts or tests;
- weaken authorization, validation, privacy, or security to make a test pass;
- publish private repository content through logs, Issues, Pages, Releases, artifacts, or
  external services;
- make production, financial, moderation, account, or legal actions outside explicit authority;
- assume that a third-party service may receive project data merely because an integration is
  technically available.

Use least privilege, synthetic data, safe failure behavior, and approved non-production
services. Validate exact targets before migrations, bulk updates, deletion, or other destructive
operations.

================================================================================
10. CONTEXT, TOKEN, AND OUTPUT EFFICIENCY
================================================================================

Reduce unnecessary context and cost by:
- searching for exact paths and symbols before opening large files;
- reading current authoritative content once;
- opening relevant ranges when the whole file is not needed;
- avoiding repeated repository inventories;
- keeping tool output focused;
- running targeted checks before broader suites;
- separating growing catalogues from their governing specifications;
- stopping repeated failed retries and reassessing the mismatch.

When a result repeatedly misses the target:
1. stop retrying the same approach;
2. identify the mismatch and authoritative requirement;
3. narrow the scope;
4. find a concrete accepted example;
5. define the relevant acceptance check;
6. retry with corrected context.

Efficiency must not remove necessary requirements, edge cases, validation, or security review.

================================================================================
11. TECHNICAL COMMUNICATION
================================================================================

Lead with the result or recommendation.

For unfamiliar concepts:
- define the term briefly;
- explain why it matters to Massage Nexus;
- provide a recommendation, not only a list of problems;
- explain meaningful alternatives and consequences in plain language.

For familiar concepts:
- use concise technical language;
- focus on project consequences;
- avoid textbook explanations and generic implementation commentary.

Do not create unsolicited implementation notes, repeat tool transcripts, or provide a formal
plan after the work is already complete.

================================================================================
12. COMPLETION
================================================================================

Before reporting completion:
- verify the requested behavior;
- inspect the final file list and diff;
- confirm unrelated user changes were preserved;
- run relevant tests and checks;
- update the authoritative specification when accepted behavior changed;
- update the Simple Checklist only when a feature is fully implemented and verified;
- record unresolved limitations or decisions honestly;
- do not commit, push, publish, or deploy unless explicitly requested.

The final response should report only:
- the result;
- important files changed;
- validation actually completed;
- unresolved limitations, risks, or decisions that matter.

================================================================================
13. CURRENT DECISIONS
================================================================================

- AGENTS.md is the active repository-agent instruction file.
- AI implements explicitly requested, ordinary in-scope repository changes directly.
- Product decisions and exploratory recommendations remain proposals until accepted.
- Additional approval is reserved for unresolved, destructive, production, external, or
  materially expanded actions.
- No unsolicited formal plan, task packet, handoff, implementation-note file, or implementation
  record is required for ordinary work.
- AI reads the smallest sufficient authoritative context.
- Working Notes remain user-controlled and outside ordinary AI work.
- Documentation rules remain authoritative in Documentation Writing and Management Standard.
- Testing rules remain authoritative in Testing and Quality Assurance Strategy and the affected
  capability specification.
- Files split by coherent responsibility, not an arbitrary line count.
- Taxonomy and field work uses repository structured-data sources and data/field_index.txt.
- Runtime AI product behavior is separate from repository-agent behavior.
- Complete means logically sufficient for the task, not maximally long.
