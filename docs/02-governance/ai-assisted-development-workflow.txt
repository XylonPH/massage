================================================================================
1. OPERATING PRINCIPLE
================================================================================

AI should reduce the user’s work rather than create repeated review cycles caused by generic,
incomplete, overlong, or poorly organized output.

The preferred first result is:
- project-specific;
- logically complete for the requested scope;
- explicit about constraints and edge cases;
- compact where explanation is unnecessary;
- detailed where implementation needs exact decisions;
- organized for manual editing;
- limited to relevant context;
- honest about unresolved decisions.

AI must not optimize for speed by producing a shallow draft that immediately requires several
more conversations.

AI must not optimize for apparent completeness by adding duplicated options, generic
strategies, unnecessary technical explanation, or unrelated “best practices.”

3. PROJECT-SPECIFIC AUTHORITY
================================================================================

Use this order:

1. current explicit user decision;
2. current accepted Massage Nexus specification;
3. current policy and security requirement;
4. current implementation contract;
5. current code and tests;
6. Shared Project Standards where not overridden;
7. older or archived material;
8. generic AI knowledge or suggestion.

Shared templates do not override a project-specific decision.

AI must not create a full generic document set merely because Shared Project Standards lists
possible documents.

Create or update only the documents that Massage Nexus currently needs.

================================================================================
4. DOCUMENT MATURITY AND VERSIONING
================================================================================

When AI creates documentation:

Use v1.00 when:
- the subject has already been thoroughly discussed;
- the decision tree is substantially resolved;
- the file records the accepted current direction.

Use v0.10 when:
- the user delegates an exploratory first draft;
- major decisions have not been discussed;
- the file is intended for later review.

When a lower-version document becomes thoroughly discussed and accepted:
- move it to v1.00;
- preserve the original Date Created;
- remove rejected proposals;
- retain accepted decisions and unresolved matters.

When splitting:
- retain the original Date Created;
- retain the exact version at the time of split;
- do not increment merely because of the split;
- use descriptive child titles;
- archive the replaced combined file.

AI must follow the Documentation Writing and Management Standard rather than infer versioning
from file age or document length.

================================================================================
5. DOCUMENT DRAFTING BEHAVIOR
================================================================================

Before drafting:
1. identify the document type;
2. determine whether the subject is thoroughly discussed;
3. read the current authoritative project files;
4. identify the exact scope;
5. identify known decisions and unresolved decisions;
6. identify cross-cutting security, privacy, language, accessibility, and data constraints;
7. determine whether an existing file should be updated instead of creating a new one;
8. determine whether a large combined file should be split.

During drafting:
- write in English for internal documents;
- use the structure appropriate to the document type;
- do not force a universal template;
- preserve exact names, codes, values, versions, routes, and filenames;
- explain unfamiliar concepts only as much as needed;
- avoid explaining basic concepts the user already understands;
- use examples where they resolve ambiguity;
- avoid generic filler;
- do not use “and others” when the scope requires actual options;
- place unresolved decisions clearly rather than inventing acceptance;
- remove obsolete concepts from the active version.

After drafting:
- check completeness;
- check duplication;
- check contradictions;
- check filename and version;
- check size and split need;
- check that the user can edit it manually;
- check that an AI can use it without reconstructing the chat.

Do not add an irrelevant section merely because it was checked.

================================================================================
7. WORKING NOTES AND AI HANDOFF
================================================================================

Working Notes:
- belong to the user;
- may be dirty, incomplete, repetitive, or inconsistent;
- are not read or edited by AI unless the user explicitly asks;
- are not an implementation authority.

AI Chat Handoff:
- is generated and updated by AI;
- preserves accepted decisions, important reasoning, unresolved questions, reusable
  deliverables, technical state, and next actions;
- does not displace current subject-specific documents;
- is maintained for continuity between chat threads or AI systems;
- normally remains outside the source repository.

AI must not treat casual scratchpad wording as a final decision.

================================================================================
8. CONTEXT SELECTION
================================================================================

The repository may contain many documents. AI must not read all of them for every task.

Default:
Read the smallest sufficient set of instructions, specifications, code, and tests needed to
complete the task safely.

Possible future routing:
- root AGENTS.md;
- closest scoped AGENTS.md;
- lightweight docs manifest;
- direct project-document references;
- affected code;
- targeted tests.

Do not recursively read:
- all documentation;
- archive;
- unrelated capability folders;
- large idea catalogs;
- Working Notes.

Expand context only when:
- a direct dependency requires it;
- a conflict appears;
- the task crosses several capabilities;
- the user explicitly requests broader review.

================================================================================
9. DOCUMENT MANIFEST
================================================================================

A future repository manifest may route AI to relevant documentation.

It should remain lightweight.

Possible fields:
- identifier;
- path;
- title;
- version;
- authority;
- summary;
- keywords;
- load_when;
- do_not_load_when;
- dependencies;
- supersedes;
- sensitivity;
- last reviewed date.

Do not require:
- an Excel document list;
- constant manual status maintenance;
- duplicate descriptions;
- a manifest before repository structure is ready.

Prefer a generated or easily maintained manifest when implementation begins.

================================================================================
10. TASK PACKETS
================================================================================

A focused AI coding task should state:

Goal
Read
Inspect
Modify
Do Not Modify
Data Impact
Security and Privacy Impact
Localization Impact
Tests
Acceptance Criteria
Known Non-Goals

Example:

Goal:
Implement protected username validation.

Read:
- Account Identity Registration and Authentication System;
- Security requirements;
- relevant translation keys.

Modify:
- username policy;
- registration validation;
- targeted tests;
- validation messages.

Do Not Modify:
- booking;
- media;
- unrelated profiles;
- unrelated database design.

Acceptance:
- blocks protected authority and brand variants;
- accepts ordinary compliant usernames;
- uses localized validation;
- targeted tests pass.

Avoid:
“Implement registration based on all our documents.”

================================================================================
11. CODE AND FILE RESPONSIBILITY
================================================================================

Split code and tests by coherent responsibility.

Prefer:
- domain-focused folders;
- one clear responsibility per class;
- business actions;
- authorization policies;
- validation requests and rules;
- separate API resources;
- focused tests;
- capability-specific translation files.

Avoid:
- one giant service;
- one giant test file;
- unrelated helper collections;
- business logic in views;
- duplicated web and API rules;
- excessive fragmentation requiring dozens of files for one simple behavior.

Refactor when a file mixes responsibilities, causes unrelated context loading, or makes
targeted testing difficult.

Do not refactor unrelated areas during a narrow task unless safety or correctness requires it.

================================================================================
12. TESTING AND OUTPUT CONTROL
================================================================================

During development:
1. run the smallest relevant test;
2. run the affected capability suite;
3. run relevant static analysis, formatting, or lint checks;
4. run the complete suite at a checkpoint and in CI.

Do not place enormous successful test output into AI context.

Summarize successful results and preserve full details only for relevant failures.

AI-feature tests use fake providers and do not spend production credits.

================================================================================
13. TOKEN AND CREDIT EFFICIENCY
================================================================================

Reduce unnecessary context by:
- searching for exact symbols before opening complete files;
- opening relevant ranges;
- using authoritative summaries;
- reading current files once;
- avoiding repeated repository inventories;
- keeping command output concise;
- running targeted tests;
- using narrow tasks;
- separating growing idea lists from governing specifications;
- archiving superseded documents;
- stopping repeated failed retries.

When a result repeatedly misses the target:
1. stop;
2. identify the mismatch;
3. narrow scope;
4. identify the authoritative file;
5. provide exact accepted examples;
6. define acceptance tests;
7. retry once with corrected context.

Do not respond to an unclear task by loading more unrelated documents.

================================================================================
14. TAXONOMY AND STRUCTURED DATA TASKS
================================================================================

AI must treat taxonomy work as structured design, not fast list generation.

For each option, consider:
- code;
- label;
- description;
- inclusion;
- exclusion;
- relationship;
- example;
- distinction from similar options.

Before adding an option:
- compare it with existing options;
- check whether it duplicates another concept;
- check whether it combines multiple concepts;
- check whether it belongs at another hierarchy level;
- check whether a new field is actually needed.

Do not:
- provide only lowercase names;
- use “and others” instead of completing accepted scope;
- add Other by default;
- create longer lists through redundant synonyms;
- confuse completeness with length.

When a taxonomy catalog grows large:
- keep the governing rules separate;
- split catalogs by family, domain, or coherent responsibility;
- preserve codes and relationships;
- keep validation against duplicates.

================================================================================
15. TECHNICAL EXPLANATION LEVEL
================================================================================

AI should adapt explanation to the user’s existing knowledge.

For familiar areas:
- use direct technical language;
- do not explain basic terms unnecessarily;
- focus on project consequences and decisions.

For unfamiliar areas:
- define the concept briefly;
- explain why it matters to Massage Nexus;
- state alternatives and consequences;
- avoid unexplained jargon;
- avoid turning the internal file into a textbook.

Public-facing guidance should explain more fully because the audience may have varied
technical knowledge.

================================================================================
16. DEVELOPMENT WORKFLOW
================================================================================

Before modification:
- confirm Git state;
- read applicable instructions;
- read only relevant specifications;
- inspect current code and tests;
- identify data, security, privacy, localization, and route effects;
- verify the task is not already implemented;
- state the narrow plan.

During modification:
- remain within scope;
- preserve naming and data conventions;
- use translation keys for user-facing text;
- keep secrets out of prompts, code, logs, and documents;
- do not weaken authorization;
- do not install unnecessary packages;
- do not expose internal technology publicly;
- use synthetic test data;
- keep changes recoverable.

After modification:
- run targeted checks;
- run broader checks at the proper checkpoint;
- remove debugging;
- update the authoritative specification when behavior changed;
- create a final-state implementation record where useful;
- update roadmap or checklist only when the milestone changed;
- report unresolved limitations honestly.

================================================================================
17. RUNTIME AI ARCHITECTURE
================================================================================

Runtime AI uses an internal provider abstraction.

Possible areas:

apps/web/app/Services/Ai/
- Client;
- Prompt;
- Translation;
- Research;
- Moderation;
- Search;
- Usage.

Flow:

Authorized action
-> permission and quota check
-> privacy and content classification
-> prompt assembly
-> provider call
-> response validation
-> human review or user preview when required
-> accepted storage
-> usage and cost record

Controllers and views do not call providers directly.

================================================================================
20. AI CRAWLERS AND REFERRALS
================================================================================

Massage Nexus prefers public discovery, citation, and referral over unrestricted bulk reuse.

Possible launch direction:
- allow ordinary search crawlers on approved public pages;
- allow approved AI search crawlers when citation and referral are beneficial;
- block training crawlers where a separate supported control exists and the final policy opts
  out;
- block private, workspace, account, booking, intake, message, moderation, evidence, and
  restricted-media paths;
- use noindex where required;
- measure approved referral traffic;
- provide controlled APIs later.

Robots directives are voluntary and do not replace security or authorization.

Exact crawler names and behavior must be checked against current official documentation before
production configuration.

25. CURRENT DECISIONS
================================================================================

- AI aims for a complete project-specific first result, not a fast generic draft.
- AI does not generate every document listed in a shared catalogue.
- Documentation versions reflect discussion maturity.
- Thoroughly discussed documents begin at v1.00.
- Exploratory documents normally begin at v0.10.
- Split documents retain the original version and Date Created.
- Internal documents are English-only.
- Public documents may be multilingual and more explanatory.
- Working Notes are user-only unless explicitly supplied.
- AI Chat Handoffs are AI-maintained continuity files.
- No universal document-section template is required.
- AI reads only task-relevant context.
- Taxonomies require codes, labels, descriptions, distinctions, and duplicate review.
- Complete does not mean long.
- Technical explanation adapts to the user’s knowledge.
- Code and tests split by coherent responsibility.
- Runtime AI uses a provider abstraction.
- Translation and research AI preserve source and review requirements.
- AI search referrals may be allowed while training crawlers may be restricted.
- Sensitive information and internal technology remain protected.

26. SUPPORT AI AND TICKET ESCALATION
================================================================================

AI supports the public Help Center and ticket workflow.

AI may answer directly when:
- approved current documentation covers the question;
- the answer is informational;
- authorization is sufficient;
- no financial, legal, security, privacy, safety, moderation, or policy discretion is needed.

Examples:
- how to add a spa;
- how to claim a profile;
- how to translate a Comment;
- where to view campaign statistics;
- how to change notification settings.

AI creates or escalates a Support ticket when:
- documented steps fail;
- a technical problem or bug is reported;
- account ownership is disputed;
- billing, refund, credit, campaign, booking, or payment action is requested;
- a complaint, appeal, privacy request, security report, or safety matter is involved;
- evidence or manual correction needs authorized review;
- no approved guidance covers the question;
- confidence is insufficient.

AI routes a matter to the project owner or another authorized decision-maker when:
- a new policy precedent is created;
- a major commercial settlement or exception is requested;
- a permanent platform-wide decision is required;
- significant legal or reputational risk exists;
- pricing, scope, or public commitments may change;
- current documentation authorizes no lower-level decision.

Escalation includes:
- concise requester need;
- relevant facts;
- attempted guidance;
- risk;
- requested decision;
- recommended options;
- privacy warning;
- linked ticket.

AI must not:
- promise refund, ranking, verification, restoration, or campaign performance;
- close a serious complaint automatically;
- create a public GitHub Issue from a private ticket;
- expose Support, advertiser, security, or private account data;
- mark a bug fixed before tested deployment.

Confirmed development work becomes a sanitized private GitHub Issue only after Support triage.

The Support knowledge base is versioned and limited to approved current guidance. Repeated
questions trigger documentation, interface, validation, automation, or product improvement.

27. GITHUB ASSISTANCE
================================================================================

AI may assist GitHub operations by:
- drafting sanitized Issues;
- proposing labels, fields, and acceptance criteria;
- linking Support defects to development work;
- creating focused branches;
- generating tests;
- drafting Pull Requests;
- summarizing Actions failures;
- updating repository documentation;
- organizing current work through GitHub Projects.

AI distinguishes:
- Support Ticket: requester case and private communication;
- GitHub Issue: internal development work;
- GitHub Project: table, board, or roadmap view over Issues, Pull Requests, and draft work;
- Pull Request: reviewed proposed source change.

AI must not publish private repository content through GitHub Pages, Releases, logs, Issues, or
artifacts.

Ordinary public Help, community Discussions, Comments, and Support remain inside Massage Nexus.

================================================================================
28. COMMENTS AND DISCUSSIONS AI
================================================================================

For Comments and Discussions, AI may:
- suggest title, category, tags, or duplicate;
- translate on demand;
- summarize long threads;
- identify possible Corrections, Support requests, or Review content;
- assist moderation triage.

AI preserves original speech, distinguishes opinion from fact, respects permissions, and does
not answer account-specific Support matters publicly.

================================================================================
29. CACHE-AWARE AI
================================================================================

Reusable AI results may be cached only when input revision, locale, permissions, provider,
model, prompt version, and safety policy make reuse valid.

Do not reuse:
- account-specific Support answers across users;
- stale policy answers;
- private prompts;
- results after source revision;
- restricted content outside its permission scope.

Caching reduces repeated translation and Support cost without making stale AI output
authoritative.
