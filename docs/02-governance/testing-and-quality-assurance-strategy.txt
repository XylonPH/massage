================================================================================
1. DOCUMENT BOUNDARY
================================================================================

Create one compact project-wide testing strategy because AI, CI, release work, and maintenance
need consistent quality rules.

Do not create one generic validation document.

Validation belongs in:
- the relevant capability specification;
- input and business rules;
- data contracts;
- client-side validation;
- server-side validation;
- database constraints;
- authorization;
- targeted tests.

Examples:
- username validation belongs in Account and Authentication;
- spa-classification dependencies belong in Spa Classification;
- booking state validation belongs in Booking;
- translation placeholder validation belongs in Translation System.

Do not create one generic implementation-strategy document.

Implementation order belongs in:
- Roadmap;
- relevant capability specification;
- narrow AI task packet;
- implementation record.

================================================================================
2. QUALITY OBJECTIVES
================================================================================

Testing should provide confidence that:
- accepted behavior works;
- invalid behavior is rejected;
- authorization is enforced;
- private data remains private;
- translations and fallbacks work;
- accessibility requirements work;
- upgrades do not break existing behavior;
- migrations preserve data;
- failures are recoverable;
- releases can be rolled back or recovered safely.

Automated tests reduce repetitive manual work. They do not eliminate manual review.

================================================================================
3. TEST OWNERSHIP
================================================================================

Each capability owns its tests.

Examples:
- Identity owns registration, login, verification, recovery, passkeys, and sessions.
- Localization owns locale selection, fallback, placeholders, and translated routes.
- Research owns provenance, freshness, contradiction, and evidence access.
- Booking owns booking states, intake, scheduling, payment, and provider operations.
- Media owns upload, transformation, access, and NSFW handling.

Cross-cutting suites may cover:
- security;
- privacy;
- accessibility;
- performance;
- integration;
- API and provider contracts;
- data and documentation integrity;
- release smoke tests;
- backup and restore.

Do not place all tests in one giant file or one universal feature suite.

================================================================================
4. TEST LEVELS
================================================================================

Unit Tests
Test one rule, class, calculation, or transformation.

Examples:
- username policy;
- age calculation;
- rating calculation;
- freshness calculation;
- translation placeholder validation.

Feature Tests
Test an application workflow through its public or internal interface.

Examples:
- register and verify;
- submit correction;
- claim profile;
- publish review;
- create booking.

Integration Tests
Test interaction between application components or external service abstractions.

Examples:
- MongoDB integration;
- email provider;
- storage;
- payment provider;
- search;
- AI provider.

API and Contract Tests
Test stable interfaces between application layers, clients, providers, and external integrations.

Examples:
- request and response shapes;
- authentication and authorization contracts;
- provider adapters;
- webhook signatures, replay handling, ordering, and idempotency;
- backward-compatible API and stored-data changes.

Data and Documentation Validation
Test machine-readable project definitions and documentation-derived artifacts.

Examples:
- PHP structure-guide syntax and field-index generation;
- JSON parsing and schema validation;
- taxonomy code and identifier uniqueness;
- documentation cross-references and repository paths;
- translation-key and placeholder completeness;
- generated-artifact drift.

Browser or End-to-End Tests
Test critical flows through a real browser where necessary.

Examples:
- registration;
- login;
- booking;
- payment;
- provider management;
- locale switching;
- restricted-content access.

Manual Exploratory Testing
Used for:
- visual behavior;
- unclear edge cases;
- mobile and responsive layout;
- usability;
- actual assistive technology;
- provider workflows;
- real-world field scenarios.

================================================================================
5. TEST ORGANIZATION
================================================================================

Organize by capability and responsibility.

Example:

tests/
- Unit/
  - Identity/
  - Localization/
  - Research/
  - Analytics/
- Feature/
  - Identity/
  - Profile/
  - Contribution/
  - Booking/
- Integration/
  - Database/
  - Email/
  - Storage/
  - Payment/
  - Ai/
- Contract/
  - Api/
  - Provider/
- DataIntegrity/
  - StructureGuide/
  - Taxonomy/
  - Translation/
- Browser/
  - CriticalFlow/
- Security/
- Accessibility/
- Performance/

File rules:
- one coherent scenario group per file;
- split when unrelated behavior appears;
- use shared fixtures and builders when several tests genuinely share them;
- avoid excessive fragmentation;
- use descriptive test names;
- keep tests readable as behavior documentation.

================================================================================
6. REQUIRED TEST CASES
================================================================================

For each rule, consider:
- valid case;
- invalid case;
- boundary;
- missing value;
- duplicate;
- unauthorized role;
- wrong owner;
- inactive status;
- expired value;
- stale revision;
- concurrent action;
- retry;
- rollback;
- localization;
- accessibility;
- privacy;
- audit;
- malicious input;
- external-provider failure.

Not every rule needs every case. The capability specification determines relevance.

Repository validation should also cover the machine-readable sources affected by a change. When
documentation, structure guides, taxonomies, translations, or generated indexes act as technical
inputs, validate their syntax, uniqueness, references, and generated output as part of the change.

================================================================================
7. CLIENT AND SERVER VALIDATION
================================================================================

Client-side validation:
- improves immediate feedback;
- may guide formatting;
- may prevent obvious mistakes;
- is never the authority.

Server-side validation:
- is required;
- enforces business rules;
- enforces authorization;
- returns localized safe errors;
- handles clients that bypass JavaScript.

Database constraints and schema validation:
- enforce uniqueness and structural integrity where appropriate (e.g., MongoDB Schema Validation);
- do not replace application validation;
- require safe error handling.

A test should prove important rules at the authoritative layer.

================================================================================
8. SECURITY TESTING
================================================================================

Test:
- authentication;
- authorization;
- account enumeration;
- rate limiting;
- session fixation;
- session revocation;
- CSRF;
- XSS;
- injection;
- insecure direct object access;
- upload validation;
- private media;
- secrets;
- production error leakage;
- protected routes;
- webhook signatures;
- payment replay;
- privilege escalation.

Security fixes receive regression tests where practical.

Do not expose vulnerability details in public test output.

================================================================================
9. PRIVACY AND DATA TESTING
================================================================================

Test:
- public versus private fields;
- provider access boundaries;
- cross-provider denial;
- data export;
- deletion;
- retention;
- consent and policy version;
- precise location;
- sensitive intake;
- analytics exclusions;
- AI data minimization;
- restricted media;
- audit access.

Use synthetic data.

Do not copy production personal data into ordinary tests.

================================================================================
10. LOCALIZATION TESTING
================================================================================

Test:
- locale selection;
- fallback;
- unsupported locale;
- translation key coverage;
- placeholders;
- pluralization;
- date and number formatting;
- right-to-left layout;
- translated email;
- validation messages;
- public metadata;
- source revision;
- stale translation;
- show original;
- on-demand user-content translation.

Critical flows must not silently fall back to incomplete or unsafe wording without the accepted
fallback behavior.

================================================================================
11. ACCESSIBILITY TESTING
================================================================================

Automated checks:
- labels;
- landmarks;
- contrast where tooling supports it;
- focusable controls and detectable focus-rule violations;
- heading structure;
- image alternatives;
- form errors.

Manual checks:
- keyboard-only operation;
- visible focus and logical focus order;
- screen reader;
- zoom;
- mobile;
- reduced motion;
- error recovery;
- dialog behavior;
- language changes.

Passing an automated accessibility scanner does not prove full accessibility.

================================================================================
12. PERFORMANCE AND RELIABILITY
================================================================================

Test when relevant:
- page load;
- search;
- large directory queries;
- booking concurrency;
- queue processing;
- media upload;
- export;
- AI timeout;
- provider outage;
- retry;
- database failure;
- cache failure;
- scheduled jobs;
- backup and restore.

Define measured targets when real usage exists.
Enable `Model::preventLazyLoading()` in appropriate test or non-production environments to detect
unexpected relationship lazy loading. It does not assert query-count stability. Use query listeners,
MongoDB command monitoring, profiling, explain plans, or explicit query-budget assertions when a
test must detect query-count, aggregation, or index-usage regressions.

Do not optimize without measurement.

================================================================================
13. MIGRATION AND UPGRADE TESTING
================================================================================

Before a schema, data, framework, or dependency change:
- backup;
- migration test;
- representative data;
- rollback or recovery plan;
- targeted regression;
- complete suite;
- staging deployment;
- monitoring.

Test:
- old data;
- missing fields;
- new defaults;
- duplicate conflict;
- index creation;
- data shape backward compatibility (especially for NoSQL migrations);
- interrupted migration;
- retry;
- rollback;
- compatibility.

================================================================================
14. EXTERNAL SERVICE TESTING
================================================================================

Use provider abstractions and fakes for ordinary automated tests.
Routine automated tests must deny or avoid real network access unless the test is explicitly marked
as an authorized sandbox integration test.

External services:
- email;
- SMS;
- payment;
- maps;
- storage;
- search;
- AI;
- social publishing;
- analytics.

Test:
- success;
- timeout;
- invalid response;
- rate limit;
- duplicate callback;
- signature;
- replay;
- out-of-order delivery;
- idempotency;
- retry;
- provider outage;
- fallback;
- cost behavior.

Do not spend production AI, SMS, email, or payment credits in routine tests.

================================================================================
15. TEST DATA
================================================================================

Use:
- Laravel Model Factories and Faker;
- fixtures;
- builders;
- synthetic identities;
- test media;
- deterministic clocks;
- fake providers;
- seeded taxonomies.

Test data should:
- be readable;
- avoid real personal information;
- include edge cases;
- identify locale and time zone;
- support cleanup;
- remain separate from production.

Use standard Laravel factories for persisted models and related model graphs when factories make the
scenario clearer and more maintainable. Explicit arrays, fixtures, or builders remain appropriate for
request payloads, malformed input, exact boundary values, provider contracts, and pure unit tests.

Test Database Isolation

The current Laravel test configuration uses an in-memory SQLite connection as the default relational
test connection and `massage_test` for MongoDB-backed application data. Tests must state or make clear
which persistence layer they exercise.

MongoDB tests must:
- fail closed unless the selected database is explicitly recognized as a test database;
- never connect to development, staging, or production data;
- create required indexes and schema validators before relying on them;
- clean every collection modified by the test or use an equally isolated database lifecycle;
- avoid assuming that clearing `user_main` also isolates other capability collections.

MongoDB-mutating tests must run serially until per-worker database names and complete cleanup are
implemented. When parallel execution is introduced, each worker must receive a unique test database
and clean it after execution.

================================================================================
16. AI ROLE
================================================================================

AI may:
- propose tests;
- generate test code;
- identify missing cases;
- run targeted tests;
- summarize failures;
- create regression tests;
- compare implementation with specification;
- suggest refactoring.

AI must not:
- remove tests merely to make the suite pass;
- weaken assertions without justification;
- replace a failing requirement with current buggy behavior;
- use production data;
- hide unresolved failures;
- claim a suite passed without execution evidence;
- hallucinate interfaces (AI must verify actual method signatures and return types before generating mocks or fakes).

Human review remains required for:
- changed business meaning;
- safety;
- privacy;
- complex user experience;
- final release decision.

================================================================================
17. TEST EXECUTION ORDER
================================================================================

During development:
1. smallest targeted unit test;
2. affected feature test;
3. affected capability suite;
4. relevant static analysis and formatting;
5. cross-cutting security or localization tests;
6. complete suite at checkpoint;
7. browser smoke test for critical flows;
8. staging verification before production.

Current local baseline for the Laravel application:
- `composer test`;
- `vendor/bin/pint --test`;
- `npm run build` when frontend assets or their dependencies are affected.

Configured CI must run the same baseline or a stricter approved equivalent. Add static analysis,
browser, accessibility, security, and performance commands only after their tools and configuration
exist in the repository.

Do not rerun the entire suite after every minor edit when a targeted test provides the needed
feedback.

When CI is configured, it runs the approved broader suite.

================================================================================
18. RELEASE GATES
================================================================================

A release should not proceed when:
- critical tests fail;
- security or privacy regression is unresolved;
- migration is not tested;
- neither rollback nor a documented forward-recovery procedure is available for affected changes;
- production debug information may leak;
- critical localized flows are incomplete;
- backup is not current;
- known data corruption exists;
- required policies are not ready;
- major accessibility blockers remain in the affected flow.

A release may proceed with accepted low-risk limitations only when:
- documented;
- user impact understood;
- no safety or security risk;
- follow-up assigned.

================================================================================
19. DEFECT SEVERITY
================================================================================

Critical
- security compromise;
- privacy breach;
- data corruption;
- payment duplication;
- core service unavailable;
- an essential flow unusable by people with disabilities without a reasonable workaround;
- prohibited content exposure;
- complete outage.

High
- broken registration;
- broken booking;
- major authorization failure;
- incorrect public provider data with substantial impact;
- major accessibility blocker.

Medium
- significant feature malfunction with workaround;
- incomplete translation;
- analytics error;
- performance degradation.

Low
- minor visual issue;
- wording;
- low-impact inconsistency.

Severity guides priority. It does not replace judgment. Priority also considers affected scope,
exploitability, exposure, reversibility, legal or regulatory impact, available workarounds, and the
time sensitivity of the affected capability.

================================================================================
20. MANUAL ACCEPTANCE
================================================================================

The user may manually review:
- visual result;
- wording;
- project terminology;
- logical classification;
- workflow suitability;
- content quality;
- business decisions.

AI and automated tests should handle:
- repetitive validation;
- regression;
- edge cases;
- authorization;
- data integrity;
- compatibility;
- routine technical checks.

Manual testing findings should become automated regression tests when practical.

================================================================================
21. TEST DOCUMENTATION
================================================================================

Do not create a large separate test plan for every feature by default.

Use:
- testing requirements inside the capability specification;
- focused test files;
- concise implementation record;
- one release checklist for a major release;
- specialized test plans only for complex or high-risk systems.

Execution evidence should record:
- commands and relevant environment or service versions;
- pass, fail, and skipped results;
- known flaky or quarantined tests;
- required manual checks;
- useful logs, screenshots, or reports when generated.

Split test documentation when one system has independent areas.

Example:
- Booking Request Tests;
- Booking Scheduling Tests;
- Booking Payment Tests;
- Booking Provider Operations Tests.

Do not create:
- one enormous Booking Testing file;
- one generic Validation document duplicating every capability.

================================================================================
22. MAINTENANCE
================================================================================

Maintain tests when:
- behavior changes;
- a defect is fixed;
- a dependency changes;
- a flaky test appears;
- runtime performance changes;
- a specification is revised;
- a provider changes.

Review:
- slow tests;
- duplicate tests;
- brittle selectors;
- outdated fixtures;
- unused helpers;
- missing critical flows;
- false positives;
- coverage gaps.

Test-count growth is not a goal. Useful confidence is the goal.

Flaky-Test Policy

- A retry may help diagnose a failure but must not silently convert an unexplained failure into a
  passing release gate.
- Quarantine requires an owner, linked issue or implementation record, failure evidence, scope, and
  review or expiry date.
- Track recurrence and fix the root cause; do not delete the test or weaken its assertions merely to
  restore a green suite.
- Remove quarantine promptly after the test is stable.

Coverage Policy

- Use coverage reports to identify important untested behavior, not as proof of correctness.
- Prefer coverage of critical paths, changed code, authorization, privacy, and failure handling over
  an arbitrary repository-wide percentage.
- Do not establish a mandatory numeric threshold until the measurement method, exclusions, baseline,
  and enforcement consequences are accepted.
- Coverage does not replace assertion quality, mutation resistance, exploratory testing, or review.

================================================================================
23. CURRENT DIRECTIONS
================================================================================

- Maintain one compact project-wide testing strategy.
- Keep validation rules inside the relevant capability specification and implementation.
- Do not create a generic implementation-strategy document.
- Organize tests by capability and responsibility.
- Split large test files and specialized plans by coherent workflow.
- Use unit, feature, integration, API and contract, data-integrity, browser, security, privacy,
  localization, accessibility, performance, migration, and manual testing as needed.
- Use synthetic data and fake external providers.
- Let AI generate and run tests, but require honest execution results and human review for
  business meaning and release decisions.
- Run targeted tests during development and complete suites at checkpoints and in CI.
- Treat automated tests as support for, not replacement of, manual product review.

================================================================================
24. CURRENT TOOLING AND OPEN DECISIONS
================================================================================

Current Implemented Tooling

- PHPUnit 12, executed through Laravel's `php artisan test` command and the Composer `test` script;
- Laravel Pint for PHP formatting validation;
- the frontend production build through `npm run build`;
- PHPUnit Unit and Feature suites;
- in-memory SQLite as the default relational test connection;
- `massage_test` as the configured MongoDB test database.

The repository does not currently contain an automated CI workflow, Pest, Laravel Dusk, Playwright,
PHPStan, or Larastan. Do not report those tools as implemented until their dependencies,
configuration, executable commands, and required CI jobs exist and pass.

Open Decisions

- CI provider, workflow triggers, supported runtime matrix, MongoDB service setup, caching, secrets,
  artifact retention, and required branch checks;
- whether PHPUnit remains the direct runner or Pest is adopted as an additional interface;
- browser testing tool and initial critical-flow coverage;
- static analysis tool, configuration level, baseline policy, and enforcement schedule;
- code coverage measurement and any future target;
- measured performance budgets and representative datasets;
- staging-data strategy;
- complete MongoDB collection-isolation and parallel-worker strategy;
- release smoke-test list;
- automated accessibility tools and supported browser matrix;
- security testing schedule, dependency scanning, and responsible review ownership.
