================================================================================
1. PURPOSE, AUTHORITY, AND REQUIREMENT LANGUAGE
================================================================================

The Edit System is the shared behavioral contract used by feature-specific forms and workflows.
It is not one giant universal form, one universal database record, or a separate role-specific
workspace. Service, establishment, practitioner, Article, Review, media, translation, taxonomy,
and other domains keep the interfaces and rules appropriate to their content.

Admin, editorial, workspace, API, import, and system mutation surfaces are Edit System clients.
They do not form a parallel authority model and must preserve the same validation, authorization,
concurrency, revision, audit, and application guarantees.

The owning feature specification determines:
- which fields exist;
- which values are valid;
- which actions are allowed;
- which source, review, verification, or publication rules apply;
- which revision collection is authoritative when a specialized revision model exists.

This document determines the shared guarantees that those implementations must preserve.

Requirement words have these meanings:
- must: accepted requirement;
- should: accepted default that requires a documented reason to override;
- may: optional supported behavior;
- example or possible: explanatory material, not an implementation commitment.

================================================================================
2. SYSTEM INVARIANTS
================================================================================

Every Edit System implementation must preserve these invariants:

- unapproved work never silently replaces accepted current data;
- authorization is enforced on the server for the action, target, field, and scope;
- a relationship, role label, byline credit, or creator reference does not independently grant
  edit authority;
- meaningful accepted changes create restorable history where the domain requires it;
- important actions create an append-only audit event;
- an editor cannot silently overwrite a newer accepted or submitted change;
- validation and sanitization run again on the server when work is saved, submitted, approved,
  applied, restored, imported, or received through an API;
- temporary autosaves do not create permanent revision noise;
- direct authority does not bypass provenance, audit, conflict detection, or high-risk safeguards;
- AI output remains attributable, reviewable, and subject to the same permissions as human work;
- record lifecycle, review decision, publication state, visibility, technical application or job
  state, and contribution status remain separate concerns;
- restoring an earlier state creates a new current revision and does not erase intervening history.

================================================================================
3. CORE TERMS
================================================================================

Record
One stored item owned by a defined collection.

Current Record
The accepted state currently used by the platform. Public visibility is a separate decision.

Edit Session
A bounded period in which an authenticated user or approved system works from a known base state.
Edit Session identity supports autosave recovery, collaboration awareness, and conflict context.
It is not itself an accepted revision or contribution status.

Temporary Autosave
Short-lived private recovery state. It is not an accepted revision or submitted proposal.

Saved Draft
A deliberate private checkpoint controlled by the owning feature.

Contribution or Change Set
A grouped proposal to add, correct, translate, enrich, close, merge, or otherwise change a record.
The shared contribution_main collection is the default proposal boundary when a specialized domain
workflow does not own the proposal.

Change Operation
One proposed field, embedded-item, relationship, or controlled record operation inside a Change
Set. A Change Set may contain several independently reviewable operations.

Direct Edit
An authorized change applied without a separate human review decision. It still requires
validation, conflict detection, accepted history where applicable, and audit.

Suggested Edit
A proposed change that cannot affect accepted data until authorized review succeeds.

Correction
A direct or suggested edit intended to fix inaccurate, outdated, incomplete, or misleading data.

Revision
A meaningful saved state used for comparison, attribution, or restoration. An accepted revision is
not the same as an audit event.

Audit Event
An immutable operational or security record that explains an important action, actor, target,
reason, result, and permitted before/after reference. Audit is not editable revision history.

Moderation Case
A policy, safety, abuse, or restricted-content case. It may refer to an edit or contribution but is
not a substitute for the proposal or revision.

Source and Field Provenance
Evidence and contextual metadata supporting a value or claim, including origin, date, verification,
freshness, confidence, contradiction, and permitted public attribution.

Conflict
A condition in which the base state used by an editor is no longer current or another authorized
decision has already changed the same work.

Application Result
The technical outcome of applying accepted operations to the target after authorization succeeds.
Application Result is not contribution status, publication state, or lifecycle state.

Review
In this document, review means an approval or verification decision about proposed work. It must
not be confused with the separate public Massage Nexus Review feature at /workspace/review.

================================================================================
4. STORAGE AND RESPONSIBILITY BOUNDARIES
================================================================================

Accepted current data
- remains in its owning target collection, such as establishment_main or article_main;
- is never reconstructed solely from audit logs at normal read time.

Drafts and proposed changes
- use contribution_main when the shared contribution workflow owns the proposal;
- may use a feature-specific draft or submission collection when specialized behavior requires it;
- remain separate from the accepted target until application succeeds.

Accepted revision history
- uses revision_main for supported records without a more authoritative specialized revision model;
- uses a feature-specific revision collection, such as article_revision, when the domain requires
  content-specific fields or decisions;
- must not duplicate full authoritative history across both generic and specialized collections
  without an explicit compatibility or indexing reason.

Audit
- uses audit_log for important immutable operational, approval, restoration, security, and system
  actions;
- links to the applicable contribution and revision where those exist;
- stores only the minimum before/after material justified by audit needs and privacy rules.

Moderation
- uses moderation_main and moderation_action for policy, abuse, escalation, and reversal cases;
- does not replace contribution review, verification, or revision storage.

Autosave
- uses short-lived private storage selected by the owning feature;
- is not written to revision_main or audit_log for every save interval or keystroke.

Current structure-guide gap
The contribution_main structure guide currently stores an opaque proposed_data object and does not
yet document base-revision concurrency fields, a stable Change Operation array, collaborators, or
application result. Until those fields are documented in the structure guide and taxonomy as
needed, selective approval and atomic multi-operation application against the shared collection
are not implementable. A single opaque proposed-data object is insufficient when reviewers may
accept some changes and reject others.

Before selective approval is implemented, contribution_main must document a stable embedded Change
Operation shape that satisfies Section 12.

5. RECORDS AND DOMAINS COVERED
================================================================================

The Edit System may support records owned by:

- Service and Service classification;
- establishment_main and establishment-owned offerings, hours, contacts, and relationships;
- practitioner_main and practitioner capabilities or offerings;
- Products, product versions, variants, sellers, and offers as defined by their specifications;
- article_main, Article bodies, revisions, tags, and related editorial records;
- review_main and rating records without allowing targets to rewrite independent Reviews;
- images, videos, documents, and other approved media collections;
- Promotions, vouchers, gift cards, and advertising records;
- translations and language-specific content;
- shared or Massage Nexus taxonomy and reference records;
- credentials, qualifications, verification, claims, and contributions;
- user-controlled profile fields;
- help, policy, announcement, legal, and platform-configuration records where authorized.

This list describes domains. A domain must use the canonical collection names and accepted field
contracts from the database architecture and structure guides before implementation.

================================================================================
6. EDITING MODES
================================================================================

Create
Create a new draft, proposal, or directly authorized record.

Direct Edit
Apply an authorized change to accepted data through the controlled application process.

Suggested Edit or Correction
Submit proposed operations for review without changing accepted data.

Translation Edit
Create or revise a target-language representation tied to a known source revision.

Bulk Edit
Apply a bounded controlled operation to selected authorized records.

Import
Create or update records from an approved structured source through validation and preview.

Merge
Combine duplicate identity while preserving aliases, redirects, relationships, evidence, and
history.

Restore or Revert
Create a new accepted state derived from an earlier accepted revision.

Archive, Restrict, Close, Remove, or Erase
Apply the distinct lifecycle or legal outcome authorized for the target; these terms are not
interchangeable.

System or API Update
Apply a controlled scheduled, integration, migration, or partner operation with an attributable
system or client actor.

AI-Assisted Draft
Generate a proposed value or structure that remains marked and subject to ordinary review.

================================================================================
7. AUTHORIZATION AND ADDITIVE RESPONSIBILITIES
================================================================================

All authenticated people remain ordinary users. Founder, administrator, editor, translator,
steward, practitioner, operator, instructor, and other responsibilities are additive permission
bundles or scoped assignments. The workspace adapts to the union of active permissions; users are
not required to switch roles or enter separate role-specific accounts.

Server authorization must consider:

- authenticated actor or approved system/client identity;
- exact requested action;
- target collection and target record;
- requested fields or embedded operations;
- active permission or role assignment;
- assignment scope, such as global, establishment, practitioner, language, editorial area, or
  course;
- verified relationship where the action requires one;
- current record and workflow state;
- source, evidence, review, or second-approval requirement;
- conflict-of-interest and separation-of-duty rules;
- recent or step-up authentication requirement;
- suspension, expiry, revocation, moderation, or incident restrictions.

An owner, investor, manager, operator, representative, or practitioner declaration never grants
workspace access by itself. A verified relationship may support a separately approved access
assignment.

Public byline credit, contributor credit, created_by_user_id, submitted_by_user_id, or other
attribution also does not grant edit permission unless the owning feature explicitly grants and
continues that permission.

Founder authority is not an unaudited bypass. Break-glass access must be time-limited, justified,
confirmed, and audited.

================================================================================
8. DIRECT EDIT VERSUS REVIEWED EDIT
================================================================================

The system decides direct versus reviewed treatment from the action, field sensitivity, target
scope, actor permission, evidence, record state, risk, and conflict—not from a role name alone.

Direct edit may be appropriate for:
- an ordinary user changing an approved self-managed biography field;
- an authorized practitioner changing permitted availability;
- an authorized operator updating permitted operational information;
- an editor revising a draft inside an assigned editorial scope;
- an administrator changing a low-risk setting within explicit permission.

Reviewed edit is normally required for:
- universal Service identity or high-impact taxonomy;
- historical, health, safety, legal, policy, or public-warning claims;
- credentials and verification state;
- profile ownership or access changes;
- merge, split, redirect, restoration, or permanent erasure;
- conflicting official and independent evidence;
- changes outside the actor's direct field scope.

The owning feature must publish a permission-and-field matrix before granting direct edit. A direct
edit still creates the accepted revision and audit event required by that feature.

Public visitors may report an error through a rate-limited abuse-resistant form where allowed, but
they cannot open private drafts, submit privileged evidence, or directly mutate a record.

================================================================================
9. EDIT FORM AND INTERACTION STRUCTURE
================================================================================

Large records should use meaningful sections, progressive disclosure, and reusable components where
a section has substantial independent rules. Components must not fragment simple fields merely to
create architectural layers.

An edit interface must provide, where applicable:

- clear identity of the target and whether the user is editing accepted data or a proposal;
- section title and logical heading hierarchy;
- visible save, autosave, submission, and conflict state;
- field-linked validation plus an accessible error summary;
- source and evidence controls where required;
- preview appropriate to public, workspace, and translated output;
- permission-aware actions without relying on hidden controls for security;
- unsaved-change and stale-base warnings;
- keyboard, touch, screen-reader, and mobile operation;
- an alternative to drag-only ordering;
- clear consequences and confirmation for high-risk actions;
- recovery guidance when an operation fails.

Feature-specific forms may look different while following these shared rules.

================================================================================
10. DRAFTS, AUTOSAVE, AND RECOVERY
================================================================================

Substantial editing should support deliberate draft saving. Autosave may be added when the content,
session length, and privacy risks justify it.

Autosave must:

- preserve the latest deliberate save and never silently replace it with an older browser state;
- identify the user, target, base revision or concurrency token, Edit Session, and saved time;
- remain private to explicitly authorized users;
- use bounded size, frequency, and retention;
- recover after interruption without publishing or submitting automatically;
- stop or move to read-only recovery after the draft is submitted or superseded;
- avoid permanent revision and audit events for every interval;
- protect sensitive fields from inappropriate browser storage or shared-device exposure.

Shared drafts must have explicit ownership or collaboration access. Public credit and operational
draft ownership remain separate.

Abandoned autosaves and drafts require documented retention and safe removal. Legal hold, active
review, dispute, or required evidence may prevent ordinary expiry.

================================================================================
11. VALIDATION AND SANITIZATION
================================================================================

Validation occurs at every trust boundary and again at final application.

Field validation includes:
- required and nullable behavior;
- length, format, range, precision, date, URL, and file rules;
- accepted language, taxonomy, and identifier values;
- rich-text and structured-content sanitization.

Record validation includes:
- relationship existence and lifecycle;
- unique slug or external identifier;
- logical date and status combinations;
- required source or evidence;
- nested item ownership and canonical field contract.

Business validation includes:
- actor authority and target scope;
- editability of the current state;
- provider or practitioner relationship boundaries;
- private-versus-public field classification;
- source revision and translation compatibility;
- conflict-of-interest and second-approval rules.

Publication validation includes:
- required public fields and approved representation;
- completed editorial, legal, verification, and moderation requirements;
- safe media, links, HTML, and embeds;
- no blocking scheduled, visibility, lifecycle, or incident condition.

Interface validation improves usability but never replaces server validation. User-facing
validation messages must use translation keys.

================================================================================
12. CHANGE SETS AND SELECTIVE APPROVAL
================================================================================

A Change Set groups related proposed operations against one target or one explicitly approved bulk
boundary. It must retain:

- target collection and target identifier, or the proposed new-record type;
- submitter and authorized collaborators;
- contribution type and reason;
- base revision reference or equivalent concurrency token;
- proposed operations in stable order;
- sources, evidence, and visibility classification;
- submission, assignment, review, decision, and application references;
- reviewer notes and submitter-visible response where appropriate;
- creation, update, submission, decision, and final-disposition times.

Each independently reviewable Change Operation must use a stable envelope inspired by JSON Patch
(RFC 6902) path addressing, not pure RFC 6902 as the complete operation vocabulary. The Massage
Nexus operation set includes at least:

- stable field or embedded-item path using JSON Pointer style (e.g., "/address_public");
- operation type from the Edit System vocabulary, such as add, replace, remove, relate, reorder,
  merge, or restore;
- base or old value where safe and necessary;
- proposed value;
- source or evidence references;
- validation and conflict result;
- reviewer decision and reason when decisions are granular.

RFC 6902 defines only add, remove, replace, move, copy, and test. Domain operations such as
relate, reorder, merge, and restore are Massage Nexus extensions and must be documented as such.

Embedded arrays must use stable item identity when items can be added, removed, or reordered. A
positional array index alone is not a safe durable target for concurrent change review.

A reviewer may accept all operations, accept selected operations, return the proposal, or reject
it. A displayed Partially Approved outcome should be derived from mixed operation decisions unless
a separately governed taxonomy field is approved. Interface wording must not create an unsupported
database status.

Selective-approval disposition defaults:

1. All accepted operations for one decision apply as one atomic unit, or as one approved
   asynchronous job with idempotent finalization.
2. Rejected operations never mutate the accepted target.
3. Rejected and returned operation payloads remain retained for authorized audit and submitter
   visibility according to privacy rules; they are not deleted merely because other operations
   were accepted.
4. When any operation is accepted and application succeeds, contribution status becomes Approved
   (APR) for the contribution decision. Mixed accept/reject is represented by per-operation
   decisions, not by inventing a Partially Approved contribution status code.
5. When no operation is accepted and the reviewer declines the proposal, contribution status
   becomes Rejected (REJ).
6. When the reviewer requires submitter changes, contribution status becomes Returned (RET).
7. A later resubmission may reuse the same contribution only when the owning feature allows it;
   otherwise the submitter opens a new contribution that may reference the prior one.

Before showing old or proposed values to a reviewer, the system must apply that reviewer's field
permissions and redact values outside the permitted scope.

Proposal confidentiality:

- drafts and unapproved proposed values are private to the submitter, explicitly authorized
  collaborators, and staff with the applicable review or investigation permission;
- rejected proposals remain visible to the submitter and authorized reviewers as retained history
  unless legal erasure or retention policy requires removal;
- staff search, export, notification, and AI processing of proposal payloads must respect the same
  field-level redaction and purpose limits as accepted-record access;
- public surfaces must not expose unapproved proposed values.

================================================================================
13. WORKFLOW AND STATUS MODEL
================================================================================

Shared contributions use status_contribution from
data/taxonomy/massage_nexus/workspace_access.json:

- Draft (DFT);
- Pending Review (PND);
- Returned (RET);
- Approved (APR);
- Rejected (REJ);
- Withdrawn (WDR);
- Archived (ARC).

Normal flow is:

Draft -> Pending Review -> Returned -> Pending Review -> Approved or Rejected -> Archived

An unapproved submitter may withdraw where legal hold, active decision, or another documented
restriction does not prevent it.

Under Review is normally an assignment or lease condition inside Pending Review, not a second
permanent contribution status. Needs Information maps to Returned for contributions. Other domains
may use the shared status_review taxonomy when that field owns their decision state.

No new workflow may invent alternate codes for an existing governed status. Existing legacy codes
must be migrated or handled through explicit compatibility logic before standardization.

These dimensions remain orthogonal:

- contribution or review decision;
- target record lifecycle;
- publication state;
- visibility scope;
- verification state;
- moderation state;
- technical application or job state (Application Result).

Application Result must be recorded separately from status_contribution. At minimum, implementations
must distinguish:

- pending application: review approved or direct edit authorized, target not yet updated;
- applied: accepted operations written and revision/audit relationships completed;
- failed application: authorized work could not be applied; accepted data must remain consistent
  with the last successful application;
- cancelled or superseded: application will not proceed because a later authorized decision or
  cancellation replaced it.

Exact Application Result field names and taxonomy codes remain an open structure decision, but the
states above are required behavioral outcomes. Approval does not imply public visibility unless the
feature's approved workflow applies and publishes in the same controlled operation. A failed
asynchronous application must not be presented as successfully applied merely because review
approval occurred.

================================================================================
14. ATOMIC APPLICATION AND IDEMPOTENCY
================================================================================

For an ordinary synchronous approval or direct edit, the following must succeed as one database
transaction or equivalent recoverable atomic unit. Because the primary data store is MongoDB, the
production environment must support multi-document ACID transactions (Replica Set or equivalent
deployment requirement documented with database architecture and operations).

1. recheck authorization, target state, and base revision or concurrency token;
2. validate the accepted operations against the current field contract;
3. apply accepted values to the target;
4. create the accepted revision where required;
5. finalize the contribution or direct-edit result and Application Result;
6. write the append-only audit event;
7. enqueue notifications or downstream work through a reliable post-commit mechanism.

Failure must not leave accepted data changed without its revision and audit relationship.
Notification or search-index failures after a successful commit must not reverse the accepted write;
they are retried or reconciled as downstream work.

Repeated requests must use idempotency protection so a retry does not apply an operation twice.
Idempotency keys should be scoped to actor or client, action, target, and a stable request or
payload identity; implementations must define storage, in-flight behavior, and retention or TTL for
those keys. A retry of an in-flight request must not create a second concurrent application of the
same accepted work.

Large imports, scheduled publication, cross-system updates, and other asynchronous applications may
use an approved job. The job must preserve the approval reference, idempotency, progress, failure,
retry, cancellation, reconciliation, and final Application Result separately from review status.

================================================================================
15. REVISION HISTORY, COMPARISON, AND RESTORATION
================================================================================

Accepted revision history must preserve enough canonical state to compare and safely restore the
target. The owning structure guide decides whether this is a full snapshot, a deterministic patch
plus base reference, or a specialized content revision.

A revision should identify:

- target and revision sequence;
- actor and source contribution or direct action;
- accepted date and summary;
- changed fields or operations;
- previous revision;
- publication and language context where relevant;
- restoration or supersession relationship.

Do not create permanent revisions for every keystroke, preview, transient formatting experiment,
or expired autosave.

Restoring an earlier revision creates a new accepted revision derived from it. It does not delete,
renumber, or rewrite later history. External effects that cannot be reversed automatically must be
identified before confirmation.

Public history may show date, safe summary, and eligible contributor credit. Internal history may
show detailed changes, reviewer, sources, permission context, and system references subject to
field-level authorization and redaction.

================================================================================
16. CONFLICT DETECTION AND COLLABORATION
================================================================================

Optimistic concurrency control is required for shared drafts, submissions, accepted records, and
review decisions that can be changed by more than one actor.

The required concurrency token is an integer version field, an accepted revision identifier, or
another monotonic server-owned token. updated_at may assist diagnostics and display but must not be
the sole concurrency token for shared writes.

The server must compare the submitted base revision or concurrency token with the current state at:

- deliberate draft save where collaborators share a draft;
- submission;
- reviewer decision;
- direct application;
- approval application;
- restoration and merge.

When the base is stale, the interface should show field-level differences and permit only authorized
outcomes:

- keep the newer current value;
- use the proposed value after renewed validation;
- manually combine;
- omit the conflicting operation;
- cancel or return the proposal.

Review queues must prevent silent simultaneous decisions through assignment leases, decision locks,
optimistic concurrency, or another approved control.

Collaboration access must be explicit. Several users may own or collaborate on one draft without
sharing credentials or changing role context. Every save, submission, and decision retains the
actual actor.

================================================================================
17. SOURCES, EVIDENCE, AND FIELD PROVENANCE
================================================================================

Important values may retain:

- source type, title, organization, and reference;
- source date and access date;
- contributor and verification method;
- freshness, last-confirmed date, and confidence;
- contradiction and source-agreement state;
- internal evidence note;
- separately approved public attribution.

Field-level provenance is especially important for operating status, hours, contacts, location,
prices, credentials, ownership, safety statements, service history, and product warnings.

The Data Research, Provenance, and Field Verification System governs evidence quality and public
disclosure. Private documents, contact details, restricted evidence, and internal confidence must
not leak through public revisions, notifications, exports, or audit snapshots.

Source URLs are untrusted input. If Massage Nexus fetches a source, the fetcher must enforce SSRF,
redirect, content-type, size, timeout, malware, and network-boundary protections.

================================================================================
18. RICH TEXT, MEDIA, TRANSLATION, AND TAXONOMY
================================================================================

Rich text
- uses the toolbar and allowed structures approved by the owning feature;
- is sanitized on the server;
- rejects scripts, event handlers, arbitrary CSS, dangerous URLs, tracking, and unapproved embeds;
- stores sanitized rendered HTML and searchable plain text where the domain requires them;
- stores a structured editor document only when the owning feature explicitly adopts one.

The Article editor may be more capable than Review, profile, help, or policy editors. A list of
possible future controls is not permission to enable unsupported markup.

Media
- retains title, caption, alt text, source, permission, copyright, AI-use label, visibility,
  moderation, crop, focal point, replacement, and archive data where applicable;
- keeps restricted files behind server authorization and protected delivery;
- does not silently erase earlier source or permission history when a file is replaced.

Translation
- shows source and target language, source revision, status, stale warning, translator, AI method,
  reviewer, preview, and fallback behavior;
- marks dependent translations stale when the source changes without deleting them;
- follows language-scoped permissions and the Translation System.

Taxonomy
- preserves stable codes, meanings, replacement relationships, translations, and usage impact;
- never reuses a retired code for a different meaning;
- requires taxonomy JSON, structure-guide, API, seed/import, filter, and client impact review;
- requires authorized review for high-impact changes.

================================================================================
19. MERGE, DUPLICATE, ARCHIVE, REMOVE, AND RESTORE
================================================================================

A merge must:

1. identify and compare candidate identities;
2. select the canonical record through authorized review;
3. resolve field conflicts and preserve source disagreement;
4. move or preserve relationships without losing independent Reviews or contributions;
5. retain aliases and old slugs;
6. create durable redirects where public identity changed;
7. preserve evidence, revisions, moderation, and audit history;
8. mark duplicate records with the governed lifecycle and canonical relationship;
9. test affected routes, search, APIs, references, and exports.

Merge is not deletion and must not erase lawful criticism or independent history.

Archive
Retains the record internally but removes it from ordinary active use.

Soft Remove
Hides the record from ordinary active use and normal queries while preserving authorized recovery
and history. Soft remove is not permanent erasure. Implementations must preserve a recoverable
hidden state; the exact mechanism (lifecycle status such as status_record_lifecycle DEL, a
framework soft-delete timestamp, or both with explicit synchronization rules) remains an open
decision and must be consistent within each domain once chosen. Soft remove remains distinct from
Archive, Restrict, Close, and Permanent Erasure.

Restrict
Limits access without asserting that the record never existed.

Close
Records that an establishment or provider ceased operating while preserving historical identity.

Permanent Erasure
Is reserved for approved legal, privacy, security, or retention reasons. It requires identity and
scope verification, legal-hold checks, high-risk confirmation, audit, and handling of backups and
downstream copies according to policy.

================================================================================
20. BULK EDIT, IMPORT, API, AND SYSTEM UPDATES
================================================================================

Bulk operations must use preview, bounded selection, per-record authorization, conflict handling,
failure reporting, resumability where useful, and an undo or recovery plan appropriate to risk.

Imports additionally require:
- size, row, file-type, encoding, and schema limits;
- malware and archive-bomb protection;
- spreadsheet-formula injection protection;
- mapping preview and validation summary;
- source-file and error-report retention limits;
- idempotent batches and reconciliation.

API and partner edits require:
- authenticated scoped client;
- target authorization;
- validation and rate limits;
- idempotency and correlation identifier;
- source client and request attribution;
- revision, contribution, and audit handling equivalent to the approved human workflow.

Scheduled processes, migrations, and integrations must not bypass the Edit System merely because the
actor is technical. System and API actors remain explicit in audit.

================================================================================
21. AI-ASSISTED EDITING
================================================================================

AI may suggest wording, normalization, missing fields, tags, translations, comparison summaries,
possible duplicates, or unsupported claims.

AI must not:

- silently publish or apply a protected edit;
- invent sources, visits, credentials, ownership, or verification;
- accuse a person or infer sensitive identity without an authorized purpose;
- change verification, access, moderation, legal, or erasure decisions without human authority;
- send restricted or personal data to an unapproved provider;
- hide the model-assisted origin of a proposal from authorized reviewers.

When AI is utilized to summarize or compare proposed changes or audit logs, the server must redact
all PII (Personally Identifiable Information) and restricted fields before constructing the prompt
for any externally hosted LLM.

AI assistance must retain the applicable method, actor or Neural Agent identity, source inputs or
references where permitted, model/provider context required by the Runtime AI System, and human
review outcome. The feature must remain operable when AI is unavailable.

================================================================================
22. NOTIFICATIONS AND WORKSPACE PLACEMENT
================================================================================

Possible notifications include draft recovery, submission receipt, assignment, return for
information, approval, rejection, withdrawal, conflict, stale source, stale translation, merge,
restore, application failure, and a direct change to a followed record.

Notifications must be permission-aware and must not expose sensitive proposed values, evidence, or
private record identity on lock screens, email subjects, or unauthorized channels. Users control
non-essential notifications.

Workspace placement follows the unified adaptive workspace:

- /workspace/contribution owns a user's contributions and suggested data edits;
- /workspace/review remains the separate public experience-Review feature;
- /workspace/article owns contributor Article drafting;
- /workspace/translation owns permitted translation work;
- /workspace/listing owns permitted establishment and practitioner listing work;
- administrative review appears in the applicable database, verification, editorial,
  translation, moderation, or other permission-scoped queue;
- /workspace/audit is a protected read-only view of permitted append-only audit events.

Navigation is generated from the user's combined permissions and assignments. The system must not
force users to choose an artificial role context before showing authorized work.

================================================================================
23. SECURITY, PRIVACY, AND AUDIT
================================================================================

All mutation endpoints require authentication where applicable, CSRF protection for browser
sessions, strict input allowlists, server authorization, validation, rate limits proportional to
risk, and safe failure behavior.

High-risk actions may require recent authentication, MFA, explicit confirmation, recorded reason,
second approval, separation of duties, or break-glass procedure.

Evidence uploads and imported files require protected storage, actual file-type validation,
malware controls, authorized delivery, and retention limits.

Audit writing and audit viewing use separate access. Audit entries are append-only and may record:

- target and action;
- actor user, system, or API client;
- result, reason, permission, and correlation reference;
- contribution, revision, moderation, verification, or application reference;
- redacted before/after reference where justified;
- time and source application;
- IP or device context only where justified by risk and retention policy.

Do not copy passwords, tokens, payment secrets, private prompts, protected intake, unnecessary
personal data, or entire sensitive records into revisions, audit events, logs, notifications, or AI
inputs.

Audit history, operational logs, security events, record revisions, moderation cases, and temporary
working notes remain distinct even when linked.

================================================================================
24. ACCESSIBILITY AND INCLUSIVE EDITING
================================================================================

Editing must support:

- complete keyboard and touch operation;
- screen-reader names, descriptions, state, and status announcements;
- logical headings, landmarks, dialogs, and reading order;
- error summaries with focus movement and field associations;
- visible focus and sufficient contrast;
- zoom, reflow, and mobile layouts without loss of action access;
- text alternatives and accessible media controls;
- alternatives to drag, hover, color-only, and pointer-only interaction;
- understandable autosave, unsaved, stale, submitting, success, and failure state;
- warning before session expiry and a recoverable timeout where security permits;
- accessible comparison of old and proposed values;
- accessible confirmation for destructive or high-risk operations.

================================================================================
25. TESTING AND ACCEPTANCE CRITERIA
================================================================================

An Edit System implementation is not complete until tests cover the applicable behavior:

- authorized and unauthorized actors at action, record, field, and scope level;
- revoked, expired, suspended, and conflicting assignments;
- accepted data remaining unchanged before approval;
- stale-base rejection and field-level conflict behavior using version or revision tokens;
- selective approval and rejected-operation isolation;
- contribution status versus Application Result separation, including failed application after
  approval;
- atomic target, revision, contribution, and audit results, including multi-document transaction
  failure paths where used;
- idempotent retry, in-flight duplicate submission, and asynchronous failure reconciliation;
- draft privacy, collaboration, autosave recovery, and retention;
- proposal confidentiality and reviewer field redaction;
- server validation, sanitization, source, file, and relationship rules;
- direct edit, reviewed edit, withdrawal, return, rejection, approval, merge, and restore;
- redaction in public history, audit, notifications, exports, and reviewer views;
- accessibility for keyboard, screen reader, touch, zoom, errors, status, and confirmation;
- regression of public routes, search, APIs, relationships, and translations.

Important user-facing workflows require manual verification in addition to automated tests. Broader
quality expectations follow docs/02-governance/testing-and-quality-assurance-strategy.txt.

================================================================================
26. CURRENT ACCEPTED DECISIONS
================================================================================

- One shared Edit System governs feature-specific forms without creating one universal form.
- Admin, editorial, workspace, API, import, and system mutation surfaces are clients of the same
  contract.
- Current accepted data remains separate from drafts and unapproved proposals.
- contribution_main is the shared proposal boundary; specialized domains may own their drafts and
  submissions when documented.
- revision_main is the generic accepted-history boundary; specialized revision collections take
  precedence for their domain.
- audit_log is append-only operational history and does not replace revisions.
- Moderation cases, verification, contribution review, publication, and public Reviews remain
  separate systems.
- Important accepted changes create meaningful restorable history.
- Autosave does not create permanent history for every keystroke.
- Direct edits still require authorization, validation, concurrency protection, and audit.
- Access is permission- and scope-based; roles and relationships do not independently grant edits.
- The workspace reflects combined permissions without role-context switching.
- Sources and provenance are supported at field or claim level where accuracy depends on evidence.
- Shared records use optimistic concurrency and cannot silently overwrite newer work.
- The required concurrency token is a server-owned version, revision identifier, or equivalent
  monotonic token; updated_at alone is insufficient.
- Change Operations use JSON Pointer-style paths and a Massage Nexus operation vocabulary. They are
  inspired by JSON Patch (RFC 6902) but are not limited to pure RFC 6902 operations.
- Selective approval applies accepted operations atomically, retains rejected operation history for
  authorized audiences, and does not invent a Partially Approved contribution status code.
- Approval application is atomic or uses an explicit idempotent asynchronous job.
- Application Result is orthogonal to contribution status; approval is not success of application.
- AI suggestions remain attributable drafts until an authorized workflow accepts them.
- Changelog, revision history, audit, moderation history, and temporary working state are distinct.
- Soft remove is recoverable hide from ordinary use and is distinct from archive, restrict, close,
  and permanent erasure; the exact storage mechanism remains open.
