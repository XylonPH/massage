================================================================================
1. PURPOSE AND BOUNDARY
================================================================================

The Admin Workspace User Interface provides authorized administrators with a consistent way to inspect queues, compare changes, review evidence, make decisions, manage operational systems, and understand the result of each action.

These tools are integrated into the unified /workspace experience. The user does not select an
"administrator" persona or switch away from their client, practitioner, operator, or other work.
Navigation and dashboard cards are composed automatically from effective permissions. Security
contexts described below concern authentication strength and risk isolation; they are not role selectors.

This document owns:

- workspace information architecture;
- navigation and page organization;
- dashboard content;
- queue, table, detail, comparison, form, and timeline behavior;
- permission-aware visibility and redaction;
- high-risk confirmation and approval interactions;
- responsive and accessible behavior;
- loading, empty, partial, failure, and degraded states;
- common administrative components.

This document does not redefine:

- which content is allowed;
- what proves verification;
- when a Review violates policy;
- final account, privacy, security, translation, payment, or legal rules;
- final database schema;
- final route implementation;
- final permission codes.

Those decisions remain in the Admin Workspace and the authoritative capability documents.

The interface is hand-built with Laravel Blade, Livewire, Alpine.js, and Tailwind CSS inside a shared branded workspace shell, rather than a packaged admin-panel framework. These UI requirements remain valid if the implementation framework changes.

================================================================================
2. DESIGN DIRECTION
================================================================================

The Admin Workspace should feel operational, focused, and trustworthy rather than decorative or consumer-oriented.

Visual direction:

- use the Ink & Ember identity without overwhelming dense administrative content;
- use Ink colors for structure, navigation, tables, and quiet surfaces;
- reserve Ember accents for primary actions, attention, and selected navigation;
- use neutral backgrounds for long review sessions;
- use compact but readable typography;
- keep dangerous actions visually distinct from ordinary actions;
- use icons only with visible text or accessible labels;
- never communicate state through color alone;
- avoid unnecessary animation;
- support light and dark color modes only when both meet contrast and data-visualization
  requirements; color mode is separate from Ink & Ember and seasonal visual themes.

The interface must prioritize:

1. current risk and urgent work;
2. assigned work;
3. actionable queues;
4. record context and evidence;
5. safe decision-making;
6. system and provider health;
7. historical and analytical information.

3. WORKSPACE SECURITY CONTEXT
================================================================================

The interface may expose three security contexts:

Standard Security Panel
- editorial;
- translation;
- stewardship;
- ordinary structured data review.

Medium Security Panel
- moderation;
- verification;
- claims;
- Support;
- restricted evidence according to permission.

Critical Security Panel
- user access;
- roles and permissions;
- system settings;
- credentials and integrations;
- backups and restore controls;
- emergency locks;
- other high-risk operations.

The current security context should be visually clear in the header and page title. Moving into a stronger context may require step-up authentication and may open in a separately configured authenticated panel. A route change alone must not be presented as proof of stronger security.

When step-up authentication expires:

- preserve safe unsaved work where possible;
- block the high-risk action;
- explain that recent authentication is required;
- return the administrator to the intended action after successful reauthentication when safe;
- never apply the action before reauthentication completes.

4. APPLICATION SHELL
================================================================================

The Admin Workspace uses a consistent shell containing:

- permission-filtered sidebar;
- workspace header;
- breadcrumb or location trail;
- page title and compact description;
- environment and security-context indicator;
- global search;
- notifications;
- current administrator menu;
- content area;
- optional contextual inspector or evidence drawer.

The shell must show a non-production environment label in development, test, and staging. Production should not use a distracting banner during ordinary operation, but critical lock, incident, degradation, and maintenance states must remain visible.

The administrator menu should include:

- display name and administrative context;
- security settings;
- active sessions and devices;
- interface language and time zone;
- accessibility preferences;
- color mode and eligible visual theme where enabled;
- exit View-As when active;
- log out.

7. ADMIN DASHBOARD
================================================================================

The dashboard is permission-personalized. It does not show blank cards for inaccessible modules.

Top status row:

- active incident or security alert;
- maintenance or lock mode;
- degraded provider or integration;
- failed backup or restore test;
- production quota or budget warning.

Primary work cards:

- assigned to me;
- unassigned allowed work;
- due soon;
- overdue;
- escalated;
- awaiting my approval;
- returned for changes.

Operational queue cards may include:

- spa and therapist submissions;
- contributions and corrections;
- Review and community moderation;
- verification and claims;
- editorial approval;
- translation backlog;
- media review;
- Support escalation;
- advertising approval;
- failed jobs and imports.

System-health cards may include:

- API and integration errors;
- AI token and cost use;
- maps and places quota use;
- email, SMS, and push delivery failures;
- storage use;
- search-index freshness;
- queue delay;
- backup status;
- unusual exports or high-risk actions.

Every card must:

- state its timeframe;
- state whether the value is a count, rate, cost, or status;
- link to the corresponding filtered page;
- show loading, stale, unavailable, and permission-limited states honestly;
- avoid implying zero when data is unavailable.

Dashboard cards should be reorderable only if the preference system can preserve an accessible default and critical alerts remain prominent.

- reported content in a safe preview;

Decision controls may include:

- no action;
- publish unchanged;
- request revision;
- approved redaction;
- label;
- limit visibility;
- remove prohibited portion;
- remove content;
- restrict or suspend account;
- preserve evidence;
- escalate;
- restore after appeal.

The interface must state separately whether an action affects:

- content visibility;
- rating aggregation;
- author account status;
- provider response;
- evidence retention;

Potentially disturbing or NSFW material uses a safe placeholder and explicit Reveal control. There is no hover reveal. The action should warn the reviewer, work with keyboard and touch, and avoid announcing hidden content to a screen reader before consent.

12. VERIFICATION AND CLAIMS
================================================================================

Verification review should display:

- target profile and verification scope;
- claimant or applicant;
- organization relationship;
- contact verification;
- submitted documents and credentials;
- evidence source and expiry;
- detected conflicts;
- previous verification decisions;
- public trust signal preview;
- limitations that will be displayed publicly.

Documents and credentials remain in a protected viewer. Download is separate from view permission and may be disabled.

Claim approval must show the exact permissions that will be granted, their context, effective time, expiry where applicable, and notification result. It must state that Reviews, ratings, universal Services, and public historical data are not transferred to the claimant.

Appeal review should be assigned to a qualified person who did not make the original decision when separation is required.

================================================================================
13. EDITORIAL REVIEW
================================================================================

Editorial pages should provide:

- assignment and workflow status;
- structured content editor or safe rendered preview;
- revision comparison;
- source list and claim verification checklist;
- author and contributor attribution;
- copyright and media permission status;
- AI-assistance record;
- sponsor and affiliate disclosure;
- legal, medical, safety, privacy, and reputational warnings;
- translation status;
- accessibility checklist;
- metadata, canonical URL, and social preview;
- scheduling;
- correction and retraction history.

Articles, News, Comics, Legal documents, Announcements, Quotes, Surveys, and Quizzes may share
list, queue, preview, scheduling, and approval components, but each page must show its actual domain
label and domain-specific requirements. The interface must not imply that these records share one
generic content collection. Announcement previews must identify the statement as first-party and
must not use an independent-News presentation.

The current Article review screen implements the first focused editorial slice: a pending count on
the Editorial landing page, an Article queue containing the latest submitted revision, a safe
rendered snapshot, reviewer note, and explicit Approve and Publish, Request Changes, and Reject
actions. It does not yet claim to implement assignment, side-by-side comparison, bulk actions,
stale-work detection, or the complete readiness checklist described above.

Quote pages should expose attribution, source, language, display window, approval, and duplicate
review. Survey and Quiz pages should expose question structure, audience, availability, translation,
accessibility, scoring or result rules where applicable, response/export controls, preview,
publication, closure, and archive.

The publication action should open a compact readiness summary. Missing mandatory approval, disclosure, source, translation, or accessibility requirements block publication rather than appearing as a dismissible warning.

18. JOBS, IMPORTS, EXPORTS, AND BULK ACTIONS
================================================================================

Job list columns:

- job or batch identifier;
- capability;
- initiator;
- status;
- progress;
- attempts;
- scheduled or started time;
- duration;
- result;
- failure class;
- permitted action.

The job detail page should show redacted parameters, correlation identifier, related records, retry history, failure details safe for the viewer, and resulting audit event.

Import flow:

1. choose approved import type;
2. upload to private quarantine;
3. validate file and scan;
4. select or confirm mapping;
5. preview affected records;
6. run dry run;
7. review warnings and errors;
8. confirm permission and impact;
9. execute bounded batches;
10. show per-record result and rollback or compensation options.

Export flow:

1. choose authorized scope;
2. explain purpose;
3. preview fields and privacy class;
4. confirm protection and retention;
5. generate asynchronously;
6. notify through a protected channel;
7. download through an expiring authorized link;
8. show expiry and deletion status.

Bulk-action UI must always show selected count, affected type, scope, preview, irreversible or compensating behavior, second-approval requirement, and partial-failure behavior. Selection across pages must be explicit and must not silently expand after filters change.

================================================================================
19. SYSTEM OPERATIONS
================================================================================

System operations should be separated into focused pages:

- configuration by capability;
- themes;
- jobs and scheduler;
- search index;
- cache;
- storage;
- logs;
- audit;
- backups and restore tests;
- maintenance and lock modes;
- security events;
- incident response.

Do not create one giant settings form.

Current implementation note: the unified workspace exposes a protected User Management entry to
the Founder through effective user_access. Account search, status controls, role assignment and
revocation, session review, and lockout safeguards are present. Full security-event history,
recent-authentication gates, and administrative audit workflows remain unfinished.

Each setting should show:

- name and explanation;
- environment;
- current effective value when non-secret;
- source of value;
- permission requirement;
- validation;
- effect and affected systems;
- whether restart, job, or cache invalidation is required;
- previous value where safe;
- rollback behavior;
- approval and audit requirement.

Secrets should be represented by status and reference metadata, never plaintext value.

Theme pages should separate draft, preview, validation, publication, scheduling, version history,
and rollback. Preview must support representative pages, supported viewport sizes, languages,
right-to-left behavior when applicable, dark or light variants where supported, missing assets,
translation completeness, and accessible contrast and focus checks.

Cache pages should default to targeted invalidation by environment, capability, target, locale,
permission scope, key family, or revision. They should show estimated scope, affected dependencies,
progress, partial failure, and audit details. Global clearing requires a stronger warning and
permission.

Statistics and rating-maintenance pages should request recomputation from authoritative records,
show a dry run or impact estimate where practical, run as idempotent jobs, and report reconciliation
results. The interface must not offer direct editing of aggregate counts as a repair method.

20. BACKUP, RESTORE, MAINTENANCE, AND LOCKS
================================================================================

Backup overview should show:

- protected backup set;
- environment;
- last successful backup;
- last failure;
- next schedule;
- encryption status;
- retention class;
- storage health;
- last restore test;
- recovery objective where approved.

Restore is not an ordinary one-click action. The UI should guide an approved runbook, require recent authentication and second approval where applicable, display the target and impact, and prefer an isolated restore test before production restoration.

Lock-mode controls should show:

- current mode;
- affected capabilities;
- public and workspace message preview;
- reason;
- start and optional expiry;
- requester and approver;
- propagation status across instances;
- rollback or unlock action;
- audit relationship.

Emergency controls should remain available only to approved break-glass operators and must not depend on the failing subsystem they are intended to control where avoidable.

21. AUDIT AND HISTORY
================================================================================

Audit search should support authorized filtering by:

- actor;
- action;
- target;
- date and time;
- result;
- permission;
- source application;
- case;
- revision;
- approval;
- correlation identifier;
- job or request identifier;
- security or privacy class.

Audit detail is read-only. There is no edit or delete action.

The interface should distinguish:

- immutable audit event;
- security event;
- operational log;
- record revision;
- moderation or Support case history.

Links may connect them, but labels and permissions must remain distinct.

Sensitive export requires reason, permission, protection, expiry, and audit. Audit search must avoid exposing secret values or unnecessary private content.

================================================================================
22. HIGH-RISK ACTION INTERACTION
================================================================================

High-risk actions must not use a generic browser confirmation.

The confirmation view should state:

- exact action;
- target and environment;
- expected effect;
- records or users affected;
- reversibility;
- known risks;
- required reason;
- recent-authentication status;
- second-approval status;
- notification result;
- planned execution time;
- rollback or compensation path.

Typed confirmation should use a meaningful target or action phrase, not a universal word such as YES.

If approval is required, submitting creates a pending approval rather than executing. The requester cannot approve their own action. Changing the action invalidates prior approval.

The execution screen must revalidate permission, target state, and approval immediately before execution.

Administrative notifications may cover:

================================================================================
24. LOADING, EMPTY, ERROR, AND DEGRADED STATES
================================================================================

Loading:
- show stable skeleton or progress without moving decision controls unexpectedly;
- identify long-running background work;
- allow safe navigation when work continues asynchronously.

Empty:
- distinguish no matching records, no assigned work, no permission, and unavailable data;
- offer a relevant action such as clearing filters when allowed;
- never imply a system is healthy solely because a queue is empty.

Error:
- show a safe message;
- preserve entered work where possible;
- provide an opaque correlation identifier;
- identify whether retry is safe;
- avoid raw exceptions, internal paths, provider secrets, and private content.

Degraded:
- identify the affected capability;
- show last successful update;
- avoid presenting stale values as current;
- disable actions that cannot be performed safely;
- provide the approved manual or fallback workflow where available.

Partial failure:
- show successful and failed records separately;
- provide per-record reason;
- expose retry only when idempotent or otherwise safe;
- never display complete success.

================================================================================
25. RESPONSIVE BEHAVIOR
================================================================================

The Admin Workspace is desktop-first because comparison, evidence, and dense operational tables require space, but it must remain usable at supported tablet and mobile widths.

Desktop:
- persistent sidebar where space permits;
- table and comparison views;
- optional context drawer;
- sticky decision area without covering content.

Tablet:
- collapsible sidebar;
- reduced visible table columns;
- detail drawer or stacked comparison;
- preserved filters and decision controls.

Mobile:
- navigation drawer;
- card or list representation for queues;
- one-column detail flow;
- explicit tabs for Current, Proposed, Evidence, History, and Decision;
- no hover dependency;
- no horizontal page scrolling for primary actions.

Complex or high-risk actions may recommend a larger screen, but viewport size is not an authorization control. If an action cannot be made safely usable at a supported width, the interface must explain the limitation without losing work.

================================================================================
26. ACCESSIBILITY AND LOCALIZATION
================================================================================

Requirements:

- keyboard access to navigation, filters, tables, dialogs, media reveal, and decisions;
- visible focus;
- skip links and landmarks;
- semantic table headers and captions;
- accessible names for icon controls;
- status text in addition to color and icons;
- live-region use only for useful status updates;
- focus restoration after dialogs and asynchronous actions;
- no hover-only information or action;
- sufficient contrast;
- text zoom and responsive reflow;
- reduced-motion support;
- accessible charts with text summaries or tables;
- warnings before restricted media;
- no hidden media content announced before deliberate reveal;
- localizable labels, validation, reasons, statuses, dates, numbers, currency, and time zones;
- source and translation language clearly identified;
- administrator-selected interface language independent from reviewed-content language.

Validation messages should identify the field, problem, and corrective action. Error summaries should link to affected controls.

30. CURRENT WORKING DECISIONS
================================================================================

- The Admin Workspace is permission-filtered and does not expose every module to every administrator.
- The interface is desktop-first but responsive and accessible at supported sizes.
- One common queue pattern supports domain-specific administrative work.
- Queue item pages separate summary, comparison, evidence, history, and decision.
- Approval, verification, claim authority, publication, and role grant are displayed as separate effects.
- Potentially NSFW media uses a deliberate accessible Reveal action and never hover reveal.
- High-risk actions use step-up authentication and purpose-built confirmation.
- External services consumed by Massage Nexus receive usage, quota, cost, error, and health views.
- Secrets are represented by status and reference metadata, never displayed plaintext after creation.
- Audit is read-only in the interface.
- Themes, Quotes, Surveys, Quizzes, Announcements, targeted cache invalidation, and aggregate
  recomputation use permission-scoped, domain-specific pages.
- The workspace shell, editorial CRUD, and moderation/system placeholders are hand-built with Blade,
  Livewire, and Alpine.js rather than a packaged admin-panel framework; Filament was evaluated and
  removed. This is a working implementation choice, not a permanent UI requirement.
