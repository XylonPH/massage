DOCUMENT DESCRIPTION
--------------------
This document defines the Translation and Localization Management System.

It separates:
- static interface translation;
- structured reference translation;
- dynamic editorial translation;
- provider-supplied text;
- legal and high-risk translation;
- user-generated content translation;
- on-demand machine translation;
- community suggestions;
- AI and manual import/export;
- coverage and quality management.

Translation does not replace the original.

Version History
---------------
- v0.40 (2026-07-19): Updated language codes in the registry to 3-letter ISO standards (e.g., 'eng', 'spa', 'jpn').
- v0.30 (2026-07-19): Simplified locale registry to support Filipino ('fil') only for the Philippine market, deprecating Tagalog, Taglish, and other regional languages (Cebuano, Hiligaynon).
- v0.20 (2026-07-18): Added semantic key context, complete locale registry direction,
  Filipino with Taglish style, on-demand discussion and comment translation, translation
  coverage, community suggestion workflow, validated AI import/export, stricter high-risk
  publication, and detailed administrative routes.
- v0.10 (2026-07-16): Established the original translation layers, source revision,
  translation status, AI labels, glossary, memory, roles, and workspace.

2. CORE TERMS
================================================================================

Language:
Recognized language.

Locale:
Language plus script, region, formatting, or jurisdiction.

Style:
Formal, conversational, legal, technical, or another register.

Source content:
Original or authoritative text.

Source revision:
Exact source version used.

Translation unit:
Smallest independently managed translatable item.

Translation revision:
Saved target-language version tied to a source revision.

Stale:
Source changed after translation.

Translation method:
Human original, human translation, professional, community, AI-assisted, AI draft, imported,
or legacy.

================================================================================

- tha;

- later justified locales.

A language record may contain identifier, key, English name, native name, public label, ISO
or custom code, public alias, script, writing direction, active status, interface status,
content status, legal status, fallback, plural rules, formats, market list, support status,
and notes.

Mixed registers like Taglish, Tagalog (tl), and regional Philippine languages (Cebuano, Hiligaynon) are deprecated to eliminate linguistic duplication.

4. SEMANTIC INTERFACE KEYS
================================================================================

Use keys based on meaning and context.

Good:
auth.login.identifier
account.registration.birth_date
booking.action.save_draft
user.action.save_favorite
research.source.last_verified
analytics.metric.profile_view

Avoid:
Log in
Save
Confirmed

Each key should preserve:
- namespace;
- default English;
- meaning;
- usage context;
- feature or screen;
- placeholders;
- plural behavior;
- allowed formatting;
- maximum length if applicable;
- accessibility note;
- sensitivity;
- source revision;
- deprecation replacement.

The same English word may require several keys when meaning differs.

================================================================================
10. USER-GENERATED CONTENT
================================================================================

Default behavior:
1. show original;
2. identify language;
3. offer Translate;
4. translate to the requested supported language;
5. label machine translation;
6. offer Show original;
7. cache using content, revision, source, target, provider, and model version;
8. invalidate or mark stale after edit;
9. allow reporting.

Do not automatically create every target-language version for every comment.

A conversation remains one thread. Translations are viewer presentations, not independent
statements by the original author.

Moderator review should be able to see original, translation, provider, report, and
uncertainty.

================================================================================
15. COMMUNITY CONTRIBUTIONS
================================================================================

Users may suggest translations only for approved areas.

Workflow:
draft
-> automated validation
-> submitted
-> review
-> revision if needed
-> approval
-> publication
-> contributor credit.

No self-approval for reputation.

Community contributors cannot directly publish high-risk text.

Language proficiency claims do not automatically grant permission. Trust is based on reviewed
work, role, and area.

================================================================================
20. SECURITY AND PRIVACY
================================================================================

Do not send unnecessary personal or restricted content to translation providers.

Sensitive translation requires:
- approved purpose;
- minimum data;
- approved provider;
- access control;
- retention rules;
- original preservation;
- human review;
- audit where justified.

Do not expose private intake, secrets, recovery codes, precise private address, confidential
evidence, or full payment credentials.

23. CURRENT ACCEPTED DIRECTION
================================================================================

- English is the source and fallback.
- Filipino (fil) is the sole local language for the Philippine market.
- Taglish, Tagalog (tl), and regional languages are deprecated.
- Static interface uses semantic keys.
- Original content remains identifiable.
- Translations are tied to source revisions.
- Discussions and comments are translated on demand.
- Community translations are suggestions until approved.
- AI translation supports draft, export, import, and review.
- High-risk text is not published from AI alone.
- Completeness is measured by area and critical flow.
- Translation management has its own workspace.
- Search uses all language forms without duplicating canonical records.
