DOCUMENT DESCRIPTION
--------------------
This document defines the working API system for Massage Nexus.

The API allows approved applications and external systems to interact with the Laravel backend without connecting directly to the database.

The website, future Flutter application, future desktop application, partner systems, and approved public developers should share the same business rules, permissions, validation, records, and audit systems.

================================================================================
1. PRIMARY GOALS
================================================================================

The API System should:

- support future Massage Nexus mobile and desktop clients;
- support approved establishment, therapist, seller, and booking integrations;
- expose selected public data safely;
- avoid duplicate business logic;
- provide stable versioned contracts;
- use controlled authentication and authorization;
- support multilingual responses;
- protect private and restricted media;
- provide predictable error handling;
- support rate limits, quotas, and audit;
- support webhooks;
- allow gradual expansion without breaking existing clients;
- keep database credentials and internal storage paths private.

================================================================================
3. WEBSITE AND API RELATIONSHIP
================================================================================

The Laravel website does not need to call its own public API for every server-rendered page.

The website may directly use:

- Actions;
- Services;
- Models;
- Repositories where needed;
- Policies;
- validation;
- application rules.

The API uses the same application layer and returns JSON.

Example:

Website request
-> Controller or Livewire page
-> Action
-> Service
-> Repository or Model
-> database
-> Blade or Livewire response.

API request
-> API Controller
-> same Action
-> same Service
-> same Repository or Model
-> database
-> JSON Resource response.

This avoids rewriting booking, permission, media, or edit rules several times.

================================================================================
18. IDENTIFIERS
================================================================================

Massage Nexus APIs use the approved cryptographically random 16-character base62 identifier when
stable record identity is required. MongoDB stores it as _id; relationship fields use the same
value through concept-specific names such as establishment_id. Predictable counter-based IDs,
UUIDs, and ULIDs are not the default Massage Nexus identifier profile.

Do not expose predictable sequential storage paths for private files.

A record may also have a readable public slug. For establishments, the canonical field name is
establishment_slug because the owning collection is establishment_main. A public website route
may use /spa/{establishment_slug} while an API, import, relationship, or audit record uses the
stable identifier.

Identifier and slug serve different purposes.

API clients should not assume that a slug never changes unless the contract promises that behavior.

================================================================================
25. IDEMPOTENCY
================================================================================

Idempotency prevents a repeated request from creating duplicate effects.

Use for operations such as:

- booking creation;
- payment action;
- order creation;
- contribution submission;
- webhook processing;
- import;
- media finalization.

The client sends a unique Idempotency-Key.

The server returns the original result when the same valid operation is repeated.

44. CURRENT WORKING DECISIONS
================================================================================

- The API is served by Laravel.
- Mobile and desktop clients do not connect directly to Azure DocumentDB.
- Website and API share Actions, Services, Models, Policies, and validation.
- First-Party App, Partner, and Approved Public APIs remain separate.
- OpenAPI is the shared contract.
- Laravel Sanctum is the working authentication direction.
- Token ability does not replace record authorization.
- Private media paths are never exposed before authorization.
- API writes use revision, validation, audit, and Edit System rules.
- Server-side AI provider keys are never delivered to clients.
- API development begins with first-party needs before broad public access.
