================================================================================
1. TERMINOLOGY AND ACCEPTED DIRECTION
================================================================================

Profile means the user's public or visibility-controlled identity and presentation.

Preference means a bounded current choice saved with user_main. Massage Nexus does not create a
separate user_preference collection. Account, appearance, accessibility, notification, privacy,
community, and reusable booking preferences are embedded in user_main.

Settings means the private interface through which the user changes preferences and controls.
Settings is not a separate data owner.

User collection means a collection whose records belong to, secure, describe, or summarize a
registered user. A user-facing aggregate may combine several collections, but the database does
not duplicate unbounded histories inside user_main.

Accepted collection names use the user_ prefix where the record is user-owned or user-scoped.
The former access_assignment collection has been migrated to user_access and removed after active
Founder access was verified. The former contribution_main
direction is renamed user_contribution. The former client_booking_default collection is retired;
bounded reusable booking preferences belong in user_main.

================================================================================
2. USER COLLECTION FAMILY
================================================================================

Core account and identity:
- user_main: account, bounded profile, bounded preferences, primary references, and summaries;
- user_contact: personal contact channels, verification, replacement, and history;
- user_address: reusable private personal and booking addresses;
- user_policy: versioned policy and consent decisions;
- user_practitioner: verified self-identity link between a user and practitioner_main.

Authority and participation:
- user_access: additive global or scoped roles and direct permissions;
- user_contribution: user proposals to add, correct, translate, enrich, or update records;
- user_saved_item: private saved targets and bounded organization metadata;
- user_follow: private-by-default following relationships;
- user_notification: one in-product notification and its inbox state; external delivery logs are
  separate.

Authentication and security:
- user_external_identity: stable external login-provider identity links;
- user_social_connection: provider-confirmed social profiles or channels eligible for display;
- user_session: active and historical authentication sessions;
- user_device: recognized-device summaries without invasive fingerprinting;
- user_mfa_method: configured multifactor methods;
- user_recovery_code: individually hashed, single-use recovery codes;
- user_passkey: WebAuthn public-key credentials;
- user_security_event: append-only security and recovery history.

Commercial and feature access:
- user_subscription: one personal subscription lifecycle;
- user_entitlement: one effective feature entitlement from a subscription, promotion, authorized
  grant, or migration;
- subscription_plan: reusable plan definition shared by user subscriptions.

Community standing:
- user_reward: immutable Nexus Point, Ember, or Inkling balance event;
- user_badge: one user badge award, progress, visibility, or revocation record;
- badge_main: reusable badge definition;
- user_rank: one immutable calculated rank interval in one versioned Rank Path;
- rank_path_main: reusable versioned Rank Path definition;
- user_reputation: one immutable reputation gain, penalty, reversal, or correction event.

Article, Review, Rating, booking, message, claim, media, and managed-record collections remain in
their owning domains. They reference user_main rather than becoming growing arrays in user_main.

================================================================================
3. USER_MAIN RESPONSIBILITY
================================================================================

user_main owns:
- account identifier, username, credential cache, account status, and membership status;
- bounded public-profile identity and privacy controls;
- private birth date and optional gender identity;
- avatar and cover media references;
- bounded account, appearance, accessibility, notification, privacy, community, and booking
  preferences;
- primary contact, booking-contact, address, and self-practitioner references;
- bounded, derived access, activity, community, subscription, policy, and security summaries.

user_main does not own:
- unbounded address, contact, policy, access, contribution, reward, badge, reputation, security,
  notification, saved-item, follow, subscription, or social-connection history;
- public professional practitioner identity, credentials, services, affiliations, availability,
  professional contacts, service areas, or media gallery;
- media binaries or processing metadata;
- article, Review, Rating, booking, claim, message, or revision history.

Derived summaries in user_main are caches. Every summary identifies its calculation time or source
sequence and can be rebuilt from authoritative records. Authorization never relies only on the
public access summary, and financial or reward reconciliation never relies only on displayed
balances.

================================================================================
4. PUBLIC USER PROFILE
================================================================================

Public route:

/user/{username}

When the profile visibility permits the viewer, the profile may show:
- approved display name and @username;
- avatar and cover image;
- biography;
- optional pronouns;
- optional gender according to the user's visibility choice;
- optional handedness according to the user's visibility choice;
- optional birthday presentation according to the user's display choice;
- official public role labels derived from active user_access records;
- optional Premium Member presentation derived from an active user_subscription;
- Nexus Points;
- featured badges and the full eligible public badge collection;
- selected Rank Path title and level;
- reputation score and label;
- published Articles, Reviews, and accepted public contributions;
- bounded public activity counts;
- provider-confirmed social connections whose display is enabled;
- the linked professional practitioner profile when applicable.

Public display-name rules:
- optional;
- minimum 2 and maximum 60 Unicode characters after normalization;
- username is the fallback when absent;
- cannot impersonate protected authority, verification, brand, or credential labels;
- is not a legal-identity or credential claim.

Biography rules:
- optional;
- when provided, minimum 20 and maximum 1,000 Unicode characters after normalization;
- plain text or a tightly controlled safe rich-text subset only;
- no executable markup, hidden tracking, impersonation, prohibited solicitation, or private data
  belonging to another person;
- subject to moderation, reporting, localization, and NSFW policy.

Official roles are not user-entered text. Only public-eligible active user_access roles may appear.
Direct permissions, internal scopes, grant reasons, and administrative notes never appear publicly.

Points, badges, selected rank, reputation, and eligible public roles are part of a visible public
profile. A private or members-only profile applies its visibility boundary to the complete profile.
Hiding a profile does not erase authoritative reward, award, rank, reputation, or access records.

Users may feature up to six eligible badges in a chosen order. The profile may provide a separate
badge-cabinet view for all current public awards. Revoked, expired, invalidated, or private awards
do not appear.

================================================================================
5. GENDER, HANDEDNESS, BIRTH DATE, AND PERSONAL PRESENTATION
================================================================================

Gender identity is optional, privacy-controlled, and stored in user_main. It:
- is not required for registration;
- does not determine eligibility, role, Rank, Reputation, default avatar, practitioner capability,
  booking eligibility, or service classification;
- is never inferred from a name, image, external identity, or practitioner record;
- may be hidden, members-only, or public;
- uses the shared gender_identity taxonomy after its option wording is reviewed for current,
  inclusive, non-duplicative meaning.

Pronouns are optional and separate from gender identity.

type_handedness is optional, privacy-controlled, and stored in user_main. It reuses the existing
RH (Right-Handed), LH (Left-Handed), AM (Ambidextrous), and UN (Unknown) controlled values used by
practitioner_main. Omission means UN. Handedness:
- is self-declared and is never inferred from writing, device use, images, bookings, or activity;
- is private by default and may be hidden, members-only, or public through
  privacy_preference.visibility_handedness;
- does not determine eligibility, accessibility, therapist selection, service suitability, Rank,
  Reputation, rewards, or authority;
- is not a medical, disability, capability, or professional-skill claim.

The exact birth_date remains a private user_main fact used for 18+ eligibility. Ordinary
self-service changes are prohibited; correction requires recent authentication, support review,
appropriate evidence when required, and a user_security_event.

The user may choose one birth-date presentation:
- HID: hidden, the default;
- AGE: derived current age;
- RNG: derived age range;
- MDY: month and day without year.

The full exact birth date is never published. Public presentation is derived at request time so it
stays correct and does not create a second birth-date value.

================================================================================
6. MEDIA
================================================================================

default_avatar_key identifies one deterministic, non-gendered Massage Nexus default avatar.
avatar_media_image_id and cover_media_image_id reference media_image records.

media_image owns the binary-storage key, provider, format, dimensions, crop, derivative, processing,
moderation, accessibility, ownership, consent, and lifecycle metadata. user_main does not store raw
binaries or permanent object-storage URLs.

Avatar and cover media require file validation, malware scanning, safe decoding, size and dimension
limits, moderation, accessible crop controls, deletion handling, and a safe fallback. NSFW avatars
and covers are prohibited. Animated avatars require a static reduced-motion poster.

A practitioner may deliberately reuse the same media_image as an account avatar and professional
profile image. Practitioner galleries remain professional media and do not become user avatars.

7. PREFERENCES EMBEDDED IN USER_MAIN
================================================================================

account_preference may contain:
- interface_language_id;
- fallback_language_id;
- preferred content languages;
- time_zone_id;
- regional date, time, number, currency, and measurement presentation.

appearance_preference may contain:
- color mode: system, light, or dark; this controls interface brightness and is distinct from
  branded or seasonal visual themes;
- text scale;
- increased-contrast preference;
- reduced-motion preference;
- eligible cosmetic selections.

notification_preference may contain:
- enabled ordinary delivery channels;
- category-level choices;
- digest frequency;
- quiet hours in the user's time zone.

Security and legally required notices cannot be fully disabled.

privacy_preference may contain:
- profile visibility;
- activity visibility;
- gender visibility;
- birth-date presentation;
- social-connection presentation;
- Premium Member presentation;
- personalization and precise-location permission choices.

booking_preference may contain one current global reusable default:
- preferred booking language;
- pressure and pressure-adjustment preferences;
- focus and avoid body areas;
- product, medium, fragrance, oil, and lotion preferences;
- attire, undressing, treatment-contact, draping, and support preferences;
- conversation and communication preferences;
- therapist-selection preference;
- relevant accessibility and assistance preferences;
- last-confirmed time.

The retired client_booking_default collection is not created. Multiple named booking presets are
not supported under the current bounded user_main direction.

Time-sensitive intake, current symptoms, injuries, pregnancy state, medication changes, safety
answers, provider observations, signatures, and consent snapshots are not ordinary preferences.
They remain protected per-booking or intake records and must be reconfirmed when required.

A booking stores the values actually selected and disclosed for that booking. Changing user_main
preferences never rewrites a historical booking snapshot.

================================================================================
8. PERSONAL CONTACTS AND ADDRESSES
================================================================================

user_contact stores one personal email, telephone, SMS, or approved messaging channel and its
verification and replacement history. It supports multiple current or historical contacts.

user_main may retain a synchronized normalized primary login-email cache for authentication and
may reference the primary personal and booking contacts. user_contact remains authoritative for
channel verification, replacement, retirement, and history.

Personal contacts are private. A booking receives a snapshot of the selected verified channel.
Changing or retiring a contact does not alter historical bookings.

user_address stores one labeled private address such as Home, Work, Hotel, or Other. It supports
multiple addresses, a default booking address, effective dates, confirmation, and protected access
instructions. Exact address and coordinates are encrypted or otherwise protected at rest.

For home service, the user explicitly selects an address. The booking stores a protected snapshot
and releases only the minimum necessary information to authorized provider roles during the
approved disclosure window. A personal address never becomes a public practitioner or establishment
address automatically.

An emergency contact describes another person and requires a separately justified,
purpose-limited design before collection. It is not silently treated as an ordinary user_contact.

================================================================================
9. USER AND PRACTITIONER IDENTITY
================================================================================

practitioner_main remains the public professional identity and may exist before registration.
user_practitioner records the verified Self link between one user and one practitioner profile.
An active Self link is normally unique in both directions and originates from an approved claim and
identity-verification workflow.

The linked self-practitioner may opt into one-way reuse of eligible public account presentation:
- avatar or cover media reference;
- approved public display name;
- biography source;
- public gender presentation;
- handedness when the user explicitly approves reuse;
- derived age or birthday presentation.

For an active verified Self link, user_main.type_handedness may be copied to the same person's
practitioner_main.type_handedness only through an explicit, audited synchronization choice. An
existing conflict is shown for confirmation rather than overwritten silently. A Steward editing
somebody else's practitioner profile cannot change the linked person's user_main value, and a
professional-profile edit cannot silently change private account identity data.

Private birth date, personal contact, personal address, recovery identity, and security data never
copy into practitioner_main. A personal contact may become a professional practitioner_contact only
through an explicit, separately verified publish action. A personal address may become a public
practice location or privacy-reduced service area only through a separate professional workflow.

A user who maintains somebody else's practitioner profile is a scoped Steward, not that
practitioner's identity. The Steward receives practitioner.manage through user_access. A factual
employment, manager, representative, or establishment relationship remains in the appropriate
relationship collection.

================================================================================
10. USER_ACCESS AND MANAGED RECORDS
================================================================================

user_access stores one additive role or direct-permission grant. It supports global, establishment,
practitioner, language, editorial, course, and other approved scopes, plus effective, expiry,
revocation, grantor, and reason fields.

Examples:
- Founder globally;
- Practitioner for the user's verified practitioner profile;
- Steward for one assigned practitioner or establishment;
- Editor for an editorial responsibility;
- temporary direct permission for one approved operation.

user_main may cache one primary public-role code and bounded managed-record counts for presentation.
Server authorization evaluates current effective user_access records, not the summary.

Managed establishments are not embedded in user_main. Factual ownership, investment, management,
employment, or representation remains in establishment_person or another accepted relationship
collection. Operational permission remains in scoped user_access. The workspace combines all
effective scopes without a role switcher.

================================================================================
11. POLICY RECORDS AND FEATURE GATING
================================================================================

user_policy stores one versioned user decision for a legal_main record: acceptance,
acknowledgment, consent grant, refusal where permitted, or withdrawal. It preserves exact document,
version, locale, action, presentation time, decision time, and approved evidence metadata.

Terms, Privacy, and other policy acceptance fields are removed from user_main as authoritative
facts. user_main may cache only policy-compliance status, outstanding required-action count, and
last check time.

legal_main identifies whether a published version requires new user action. When required action
is outstanding:
- public browsing remains available;
- login may complete;
- policy review, logout, support, privacy-rights, and essential security routes remain available;
- ordinary workspace entry and protected mutations are blocked until the required decision;
- rejection or refusal follows the specific policy's documented consequence.

Cookie choices are distinct from Terms acceptance. Essential authentication and security cookies
are not optional. Nonessential analytics, advertising, campaign, or third-party purposes remain
disabled until permitted. Anonymous choices remain browser- or consent-token scoped; authenticated
current choices are cached under user_main privacy_preference so they follow the user across
devices, while versioned grant, refusal, and withdrawal history belongs to user_policy when legally
and technically appropriate.

================================================================================
12. EXTERNAL IDENTITIES AND CONNECTED SOCIAL ACCOUNTS
================================================================================

user_external_identity stores a stable provider-subject link used for login or account linking.
Email text alone never merges accounts. Priority providers remain Google, Apple, Microsoft, and
Facebook, followed only by approved market-relevant providers.

user_social_connection is separate because a login identity is not necessarily a public Facebook
page, Instagram profile, YouTube channel, X account, or other creator profile. A social connection:
- is established through the provider's supported authorization or verification flow, not an
  unrestricted URL field;
- stores the stable provider subject or channel identifier and provider-returned presentation;
- supports multiple eligible channels where the provider permits it;
- is private by default;
- appears publicly only when the user enables display;
- can be disconnected without deleting historical security evidence;
- never grants Massage Nexus authority, verification, reputation, or professional credentials.

Access and refresh credentials are encrypted or held through an approved secret store and never
rendered in profile, logs, exports, or ordinary administration screens. Revocation, expiry, scope,
and last synchronization are tracked.

================================================================================
13. SESSIONS, DEVICES, MFA, PASSKEYS, AND SECURITY HISTORY
================================================================================

user_session stores one active or historical authenticated session, including user, optional
recognized device, authentication method, safe approximate client context, creation, last activity,
expiry, and revocation. Raw reusable session secrets are never stored in readable form.

user_device stores one user-recognized device summary. It does not authorize invasive or covert
fingerprinting. A user may name, recognize, distrust, and remove a device.

user_mfa_method stores one configured second-factor method. TOTP secrets are encrypted and never
returned after confirmation. SMS is not the sole strong recovery method.

user_recovery_code stores one individually hashed, single-use recovery code and its consumption or
revocation state. Plain recovery codes are shown only once at generation.

user_passkey stores WebAuthn credential ID, public key, signature counter, authenticator metadata,
name, transports, and lifecycle. Massage Nexus never stores biometric templates.

user_security_event stores append-only login, logout, failed authentication, password reset,
contact change, session revocation, MFA, passkey, recovery, external-identity, suspicious-activity,
and privileged support events. Sensitive values and reusable secrets are excluded.

user_main may cache active-session count, passkey count, MFA-enabled state, and last security-event
time. These values are not authoritative.

================================================================================
14. SAVED ITEMS, FOLLOWING, AND NOTIFICATIONS
================================================================================

user_saved_item stores one saved supported target, optional private note, folder label, and
lifecycle. Saved items are private by default. Shared lists require a separate accepted sharing
design.

user_follow stores one user-to-target following relationship and a per-follow notification choice.
It is private by default, including following another user. Public disclosure requires explicit
later opt-in and never changes existing records automatically.

user_notification stores one in-product notification and its unread, read, archived, or expired
state. Email, SMS, and push delivery records remain in their delivery domain. Notification payloads
use safe templates and references and do not copy private intake, exact residential address,
recovery secret, or other sensitive content into unsafe channels.

Preferences for these systems remain bounded in user_main; the growing records do not.

================================================================================
15. SUBSCRIPTIONS, PREMIUM PRESENTATION, AND ENTITLEMENTS
================================================================================

An ordinary registered user may remain on the free plan or purchase a personal premium plan.
Premium is a commercial membership state, not a workspace role, Rank, Reputation level,
verification, credential, or authority grant.

subscription_plan defines reusable plan identity, audience, billing interval, price direction,
limits, included entitlement codes, and public label eligibility. user_subscription records one
user's subscription lifecycle. user_entitlement records one effective feature benefit and its
source, limit, use window, and revocation.

Possible personal premium benefits include:
- ad-free browsing;
- expanded saved lists and alerts;
- enhanced comparison, travel, history, and budgeting tools;
- increased personal profile customization or media limits;
- approved premium AI allowance;
- advanced but non-safety-critical booking convenience;
- early access to selected nonessential features.

Essential discovery, safety information, ordinary Reviews, account security, privacy controls,
policy access, correction rights, and ordinary booking safety controls remain outside a paywall.

An active subscriber may choose to show an approved Premium Member label. Payment does not increase
Nexus Points, badges, Rank, Reputation, rating weight, organic visibility, moderation power, or
workspace permission.

================================================================================
16. REWARDS, BADGES, RANKS, AND REPUTATION
================================================================================

user_reward is an immutable reason-based event for earning, spending, expiring, reversing, or
adjusting Nexus Points, Embers, or Inklings. user_main caches lifetime points and current currency
balances with a calculation time.

badge_main owns reusable badge definitions. user_badge owns one user's award, progress, featured
display choice, expiry, revocation, and source eligibility. A badge never grants authority by
itself.

rank_path_main owns versioned path and level definitions. user_rank stores an immutable calculated
interval with path, level code, qualifying point total, calculation version, effective time, and
end time. user_main caches the selected path and current level for primary public presentation.

user_reputation stores immutable positive, negative, reversal, and correction events. user_main
caches the current bounded 0-10,000 score, level code, status, and calculation time.

Reward and reputation events include idempotency keys, badge awards use a user-and-badge uniqueness
rule, and rank calculations use a calculation key so approval or recalculation retries do not apply
twice. Rejected, fraudulent, duplicate, self-dealing, revoked, quarantined, or otherwise ineligible
activity does not produce a lasting positive event.

================================================================================
17. USER CONTRIBUTIONS, EDITS, AND ACTIVITY
================================================================================

user_contribution records one user proposal and its target, proposed values, declarations, review
state, reviewer, and decision. It replaces the contribution_main name.

Ordinary proposal flow:

user_contribution -> review -> accepted target update -> record_revision -> audit_log
                  -> eligible user_reward/user_reputation/user_badge/user_rank updates

An authorized Steward or editor may create a direct accepted record revision where the Edit System
permits it. The revision and audit event still identify the actor. Privileged maintenance receives
rewards only when the governing reward rule explicitly allows it.

user_main activity_summary contains bounded, rebuildable counts, not record-ID arrays. Initial
counts may include submitted and published Articles, submitted and published Reviews, submitted and
accepted contributions, approved media, Ratings, Comments, and managed-record totals.

As the related features are implemented, supported factual totals may also include establishments
or practitioners added, discussions participated in, approved edits, reports, translations, events,
quizzes, polls, completed bookings, and completed purchases. Only bounded totals needed for current
profile or workspace presentation belong in user_main; detailed and unbounded history remains in
the authoritative domain collections.

These counts are factual statistics, not reward balances or trust scores. A qualifying action may
increase a contribution count without earning Nexus Points, and it may earn a reward without being
classified as a contribution. Bookings, purchases, events, polls, quizzes, and campaigns are
participation statistics unless their governing specification explicitly classifies them as
contributions. Nexus Points, Embers, Inklings, badges, Rank, Reputation, and user_access therefore
remain separate from activity_summary.

Created record, credited author, submitter, reviewer, publisher, and last editor are separate facts.
Public activity shows only eligible published attribution and never exposes private drafts,
rejected submissions, internal review, or hidden identity.

================================================================================
19. PRIVACY, SECURITY, LOCALIZATION, AND ACCESSIBILITY
================================================================================

- Private identity, contact, address, security, policy, booking, and payment data never becomes
  public merely because a profile exists.
- Public fields use safe private defaults and explicit audience labels.
- Sensitive changes require recent authentication and user_security_event creation.
- Exact addresses, contact values, TOTP secrets, provider tokens, and protected recovery material
  are encrypted, hashed, tokenized, or stored through an approved secret system as appropriate.
- Public profile content and social presentation are localized and accessible.
- Profile images include accessible text or are correctly marked decorative.
- Animated media honors reduced motion.
- Forms are keyboard accessible and announce validation, visibility, and saved state.
- Data export and deletion apply collection-specific retention, legal-hold, attribution, fraud,
  and transaction rules rather than deleting authoritative history blindly.

20. IMPLEMENTATION AND MIGRATION ORDER
================================================================================

Documentation and structure guides establish the accepted target state. Runtime authorization now
uses user_access exclusively after verified migration and removal of access_assignment. Optional
profile fields and bounded preference groups are stored sparsely instead of as null values or empty
arrays; username is the public route key and is not duplicated in user_slug. Public profiles,
embedded profile/privacy/appearance/notification preferences, server-backed color-mode selection, session review and revocation, coarse
recognized-device summaries, and protected administrator user/role/session surfaces are implemented.
The runtime still uses contribution_main and Terms fields in user_main.

Safe implementation order and current progress:
1. Add target collections, indexes, taxonomies, validation, authorization, and tests.
2. Backfill user_contact, user_policy, and user_access from current user_main and legacy records.
3. [Implemented] Migrate access_assignment to user_access, verify active Founder access, and remove
   the legacy collection.
4. Dual-read or migrate contribution_main to user_contribution.
5. Move reusable booking preferences into bounded user_main.booking_preference while preserving
   booking snapshots and protected intake.
6. Add policy gating and legal_main reacceptance metadata.
7. [Partially implemented] Public profiles are active. Media, social connections, and
   self-practitioner linking remain pending; profile and cover media are deliberately outside this change.
8. [Partially implemented] Session and coarse device management are active. MFA enrollment,
   recovery codes, passkeys, recent-authentication gates, and security-event history remain pending.
9. Add rewards, badges, Rank, Reputation, subscriptions, and entitlements with authoritative event
   records and rebuildable summaries.
10. Add saved items, following, and notifications.

Never rename or remove a live collection before the running application, migration, rollback,
indexes, and active-account verification are ready.
