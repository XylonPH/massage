This document defines the intended repository, application, documentation, source-asset, API-contract, localization, storage, test, and code folder structure for Massage Nexus.

Confirmed repository information:

- repository root folder name: massage
- GitHub repository: https://github.com/XylonPH/massage
- local PHP installation path: C:\php
- Laravel framework (v13.20.0) is initialized under apps/web/ and integrated with MongoDB.

It is deliberately expanded so human and AI contributors know where new work belongs before creating arbitrary files or oversized all-purpose files. Empty folders do not all need to be created immediately, but new files must follow this structure unless an accepted decision updates this document.

================================================================================
1. STRUCTURAL PRINCIPLES
================================================================================

1. One Git monorepository contains shared documentation, contracts, source assets, and separately buildable applications.

2. Different programming languages may coexist because each application has its own folder, dependency files, commands, tests, and releases.

3. The Laravel installation lives under apps/web/, not in the repository root.

4. The future Flutter application lives under apps/mobile/.

5. The Massage Nexus Desktop Application lives under apps/desktop/ when desktop implementation
begins. It is a general desktop application, not an administration-only dashboard. Features
remain permission-aware, so authorized administrative tools may still appear within it.

6. Shared decisions, API contracts, taxonomy contracts, localization keys, and brand sources exist once at the repository root.

7. Only apps/web/public/ is directly exposed by the web server.

8. Private files, secrets, planning documents, source code, and editable brand masters never belong under a public web directory.

9. A file should have one clear main responsibility.

10. AI contributors must search the existing structure before creating a new folder or a second competing implementation.

11. Refactored code replaces the earlier code in its correct location. Do not create folders named old, new, refactored, final, latest, or fixed for source code. Git preserves earlier versions.

12. Temporary work belongs only in approved temporary locations and must be removed or consolidated after acceptance.

================================================================================
2. FILE-SIZE AND SPLITTING RULES
================================================================================

These thresholds apply to ordinary hand-written application code, not project documentation.
Documentation follows docs/02-governance/documentation-writing-and-management-standard.txt.

Target size for ordinary hand-written code:
Under 250 lines where practical

Review for splitting:
300 to 400 lines

Requires a brief reason during task or code review, not a separate implementation-note file:
More than 500 lines

These are review thresholds, not automatic failure rules. A cohesive file may remain longer when
splitting would obscure its purpose, duplicate state, or make the code harder to follow. Generated
files, localization catalogs, configuration data, migrations, tests, and narrowly focused reference
files may also legitimately be longer.

Split a file early when it contains several independent concerns, such as:
- page layout and database queries;
- profile header and all profile tabs;
- unrelated forms;
- unrelated API resources;
- several large tables;
- several independent JavaScript features;
- several business operations;
- authorization and business logic;
- rich-text editor configuration and unrelated page behavior.

Do not split merely to create dozens of tiny files with no meaningful responsibility. A useful file has a precise name and a clear reason to exist.

The purpose of this rule is to reduce the amount of unrelated code that a human or AI contributor
must read for a focused change. Split by responsibility and dependency boundary, not merely by line
count. A refactoring must preserve behavior and place the extracted logic under a precise name.

4. SHARED ROOT FOLDERS
================================================================================

apps/
Contains separately buildable applications.

assets/source/
Contains editable master artwork and media sources. Examples include Illustrator files, original PNG references, layered design files, source audio, and source video. These are not automatically deployed.

assets/export/
Contains approved exports prepared for specific uses. Website copies are still placed under the web application's resources when needed.

contracts/api/
Contains the agreed API contract shared by Laravel, Flutter, and any later desktop client.

contracts/taxonomy/
Contains machine-readable shared taxonomy contracts or schemas when several applications need to understand the same stable values.

contracts/localization/
Contains shared localization keys, locale mappings, and exchange formats. It does not replace each application's native translation files.

contracts/event/
Contains shared event or webhook definitions.

docs/
Contains current internal project documentation. Public help content may be drafted here but is not automatically published merely because it exists in this folder.

tools/
Contains project-maintenance scripts. Application runtime code does not belong here.

.github/
Contains GitHub automation, issue forms, pull-request templates, and workflow files.

Actions/
Contains one clearly named operation, such as PublishArticle, CreateBooking, MergeService, or ApproveContribution. Use an Action when a single task is called from more than one interface or deserves isolated testing.

Console/
Contains terminal commands used by developers, scheduled jobs, or operations staff.

Data/
Contains temporary, labeled packages of information passed between parts of the program. A Data object is not automatically a database record.

Enums/
Contains small technical fixed choices that rarely change. Multilingual expandable taxonomies do not normally belong here.

Events/
Represents something that happened, such as BookingConfirmed or ArticlePublished.

Listeners/
Responds to an Event, such as sending a notification after a booking is confirmed.

Exceptions/
Contains project-specific errors that need clearer handling.

Filament/
Contains administration resources, custom pages, and dashboard widgets used by the browser-based workspace.

Http/
Contains website and API request handling.

Jobs/
Contains background work, such as media processing, email delivery, search indexing, or AI summary preparation.

Livewire/
Contains server-driven interactive pages and components.

Models/
Represents records stored in MongoDB or Azure DocumentDB.

Notifications/
Defines notices sent through website, email, SMS, or push channels.

Policies/
Answers whether a particular user may perform an action on a particular record.

Providers/
Starts or registers application services and framework behavior.

Repositories/
Contains repeated or complicated data-reading and data-saving operations. A Repository is like a librarian that knows how to find records. It is not required for every simple Model query.

Rules/
Contains reusable validation checks that Laravel's ordinary validation rules cannot express clearly.

Services/
Contains larger reusable business or integration work, such as booking coordination, media delivery, moderation, AI calls, and search.

Support/
Contains small reusable tools shared across several modules, such as money formatting, identifier generation, HTML sanitization, or date handling. It must not become a miscellaneous dumping folder.

ValueObjects/
Contains small meaningful values with their own rules, such as Money, DateRange, GeoPoint, or LanguageCode, when justified.

Receives a request and returns a response. Controllers should be thin. They should not contain entire business workflows.

Checks submitted values, such as required fields, valid ratings, allowed file types, dates, and taxonomy codes.

Resources/
Turns Models or prepared data into controlled JSON for APIs. These are not image resources.

Media Controllers/
Deliver public, private, or restricted files according to storage and authorization rules.

10. ACTIONS, SERVICES, AND REPOSITORIES
================================================================================

Example request flow:

Public or API request
  -> Controller or Livewire page
  -> Action
  -> Service
  -> Repository or Model
  -> Database or external system

Not every request requires every layer.

Example:

CreateBooking Action
- receives CreateBookingData;
- asks BookingService to apply booking rules;
- uses repositories or Models to read schedules and save the booking;
- sends an Event;
- a Listener sends notifications.

Repository meaning:
A repository is a place where things are stored or collected. In Git, the whole project is a repository. In application code, a Repository class is a focused place for complicated record access.

Use a Repository when:
- the same complex query appears repeatedly;
- document-store query logic becomes difficult to read;
- search can use different backends;
- testing benefits from replacing a data source;
- a feature reads several collections in a consistent way.

Do not create a Repository merely to wrap one simple Model call.

Examples:

Support/Formatting/MoneyFormatter.php
Formats 1500 PHP as a localized display such as PHP 1,500 or the approved currency style.

Support/Identifier/OpaqueIdentifierGenerator.php
Creates the approved cryptographically random 16-character base62 internal record identifier.
MongoDB collections use this value as _id, and related records use the same value in their
concept-specific *_id references. It replaces counter-based sequential identity.

Internal identifier, public slug, and human reference number remain separate:
- establishment_main._id is the stable random internal identifier;
- establishment_slug is the readable public route value, such as lotus-wellness-spa;
- a booking, ticket, invoice, or other human-facing reference number is created separately only
  when the feature requires one.

The public spa route may use /spa/{establishment_slug}. APIs and imports may use _id or the
relationship name establishment_id when stable record identity is required. An establishment
title is not an identifier and must not be used to decide uniqueness.

Support/Sanitization/HtmlSanitizer.php
Cleans rich HTML before public display.

Support/Time/Clock.php
Provides a consistent current-time source for tests and runtime logic.

Support/Concerns/HasPublicSlug.php
Contains a narrowly reusable behavior used by several Models.

A Support item must be small, reused by several modules, and precisely named. Business logic for one feature stays in that feature's Action or Service folder.

Support/Demo/
This is the approved temporary location described in Structural Principle 12 (temporary work). Use it only for static, clearly fictional, render-time placeholder content that stands in for a feature's real data layer before the matching Model, Repository, or Service exists, such as Support/Demo/SampleContent.php for the public homepage and spa profile pages before establishment_main and related MongoDB collections exist. A Support/Demo/ class must not be reused as the eventual real data source; delete it and update the calling Controllers or Livewire pages once the real Model-backed data path is built. Do not add unrelated reusable utilities to Support/Demo/.

19. PRIVATE AND RESTRICTED MEDIA DELIVERY
================================================================================

Suggested code:

app/Http/Controllers/Media/
- ShowPublicMediaController.php
- ShowPrivateMediaController.php
- ShowRestrictedMediaController.php

app/Policies/
- MediaPolicy.php

app/Services/Media/
- PublicMediaService.php
- PrivateMediaService.php
- RestrictedMediaService.php
- TemporaryMediaUrlService.php

The browser must not receive a private file location before authorization.

Correct flow:

1. Browser requests a protected media record.
2. Laravel identifies the user or API client.
3. MediaPolicy checks permission.
4. Laravel streams the file or creates a short-lived protected URL.
5. Unauthorized users receive a denied response.
6. Sensitive access may be recorded.

Do not:
- put the original under public/;
- hide it only with CSS;
- place the real URL behind a lock overlay;
- expose the path in JavaScript;
- depend only on unpredictable filenames;
- allow directory browsing;
- cache restricted content publicly.
